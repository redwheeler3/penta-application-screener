# Penta Application Screener Specification

## Purpose

The Penta Application Screener helps screen 300+ housing co-op applications for Penta Housing Coop. It imports application responses from a Google Sheets response spreadsheet in the Penta Google Drive folder, applies deterministic hard filters, uses AI-assisted review for essay answers, and produces a committee-ready report for MOMI (Move In Move Out).

The project is also a deliberate learning and portfolio project for Jeff to build practical expertise in AI product management, agentic workflows, evals, cost-aware model use, human-in-the-loop product design, and AI-assisted software delivery. The code may eventually be made public as part of Jeff's AI product management portfolio, so the implementation should be understandable, well-documented, and credible as a real product artifact while preserving applicant privacy.

## Primary User

The primary user is Jeff. The output audience is MOMI, who need a clear shortlist of applicants recommended for the interview stage, with justification.

## Product Concept

The screener proceeds in phases:

1. Import and normalize application data from Google Sheets.
2. Apply deterministic hard filters without AI.
3. Use AI agents to evaluate essay-question answers and surface patterns.
4. Ask the user targeted questions based on those patterns.
5. Re-rank or narrow candidates after each user answer.
6. Continue until the candidate list reaches a user-approved threshold for manual review.
7. Produce a MOMI-ready report with recommended interview candidates and justifications.

The screener should eventually support multiple MOMI committee members running their own screening sessions independently. Each member may value different criteria. The app must preserve and summarize each member's criteria, question answers, shortlist, and rationale so MOMI can compare both applicant recommendations and the values/criteria behind those recommendations.

## Deterministic Eligibility Rules

### Rules Engine Architecture

The screening rules system is a configurable rules engine. Each rule is a discrete, named validation that produces a binary outcome: the application is either `eligible` or `filtered_out`.

Each rule has:

- **ID**: machine-readable slug (e.g. `owns_real_estate`, `child_age_over_max`)
- **Display name**: human-readable label shown in the admin UI (e.g. "Real estate ownership")
- **Description**: explains what the rule checks and why
- **Outcome**: `filtered_out` (the only outcome — any rule that fires disqualifies)
- **Parameters**: configurable thresholds or values (e.g. income min/max, min/max children, max child age, max pets). Not all rules have parameters.
- **Enabled**: toggle on/off per screening configuration

Rules are stored in the database as part of admin settings. The Admin settings UI shows the full rule list with toggles and parameter inputs. Disabled rules do not run during screening.

Adding a new rule requires code (a rule function that takes normalized application data and returns pass/fail with a reason). Once the code exists, the rule appears in the admin UI and can be configured. The goal is that the rule logic is simple enough to add and that the admin can control which rules are active and what thresholds apply without code changes.

Rules run in a defined order. An application that fails any enabled rule is `filtered_out`. An application that passes all enabled rules is `eligible` and proceeds to AI screening.

### Rule Catalog

The following rules should be implemented. Each rule is listed with its default outcome and parameters.

**Household composition rules:**

| Rule ID | Description |
|---------|-------------|
| `child_count_mismatch` | Declared child count does not match the number of complete child detail blocks. A complete block = first name + last name + age all filled. |
| `too_few_children` | Household child count is below the configured minimum. Parameter: min_children (default 1). Moved off the form so the form can drop branching validation. |
| `too_many_children` | Household child count is above the configured maximum. Parameter: max_children (default 4). Moved off the form likewise. |

**Age rules:**

| Rule ID | Description |
|---------|-------------|
| `child_age_over_max` | Any listed child is older than the configured maximum child age. Parameter: max_child_age (default 17 — i.e. the form's "children under 18"). Tunable for co-ops housing older dependants. |
| `applicant_under_min_age` | Applicant age is under the configured minimum adult age. Parameter: min_adult_age (default 18). |
| `co_applicant_under_min_age` | Co-applicant age is under the configured minimum adult age (default 18). Indicates the co-applicant may actually be a child, not an adult household member. |
| `child_age_exceeds_parent` | Any child's age is older than the applicant's or co-applicant's age. Data entry error. (Distinct from `child_age_over_max`: this is a sanity check against the household's own adults, not the policy ceiling.) |

**Financial rules:**

| Rule ID | Description |
|---------|-------------|
| `income_below_range` | Household gross income is below the configured minimum. Parameter: min_income (default $70,000). |
| `income_above_range` | Household gross income is above the configured maximum. Parameter: max_income (default $150,000). |
| `income_arithmetic_mismatch` | Applicant income + co-applicant income does not exactly equal the stated household total. No tolerance — any discrepancy is flagged. |

**Property rules:**

| Rule ID | Description |
|---------|-------------|
| `owns_real_estate` | Applicant owns real estate. |

**Data integrity rules:**

| Rule ID | Description |
|---------|-------------|
| `negative_number` | Any whole-number-validated field (age, income) contains a negative value. The form allows negative integers but they are clearly data entry errors. |
| `future_employment_start` | Employment start date is in the future. |
| `co_applicant_incomplete` | Some co-applicant fields are filled but others are blank (partially filled). Nothing or everything is fine; partial is not. |

### Rule Behavior Notes

- Living at the current address for less than 2 years is not disqualifying and does not trigger any rule.
- Applicants outside Vancouver, BC, or Canada are eligible.
- Applications should be complete at submission time. The screener does not create applicant follow-up workflows.
- Applicants with an application already on file are not treated differently.
- Email-list signup date and notification history must not influence screening.
- For child age calculations, use age on the move-in date when a date is needed. A child turning 18 shortly after move-in does not matter.

### Application Status Model

Each application has a single mutable **status** with exactly two values:

- `eligible`: in the running, proceeds through screening
- `ineligible`: not in the running

Status is set by an actor, recorded in **`status_source`**:

- `untouched`: no actor has acted on it — it passed the rules and either AI has not run or AI did not flag it. The default for a clean eligible application.
- `rules`: the deterministic filters set it `ineligible` (high trust)
- `ai`: the AI screening pass set it `ineligible` (lower trust — this is the "needs review" bucket)
- `human`: a person set the status, in either direction

Only an actor that *acts* stamps itself. Rules passing an application through, or AI declining to flag it, leaves it `untouched` — they do not "decide" eligibility, they hand it to the next step. Only a human can move an application from `ineligible` back to `eligible` (or the reverse).

There is no third status. The UI surfaces the `status_source = ai` group as an "AI Flagged" view, composed client-side as a filter over the real columns. This keeps status binary while distinguishing high-trust deterministic exclusions from AI exclusions. The labeling is deliberately factual ("AI Flagged" — what happened), not prescriptive ("Needs Review" — what the user must do): whether to review an AI exclusion, and which flags matter, is the human's judgment, not the system's. The backend never names these views; it returns counts and filters keyed by the real `status` and `status_source` columns. When the two filter groups combine, their counts are faceted — each group's counts reflect the other group's active filter (plus search) — so impossible combinations read zero rather than a misleading total.

**The "why" is kept separately as immutable records**, never mutated by a human:

- deterministic **filter reasons** (e.g. `Household gross income ($164,000) is above $150,000.`)
- **AI screening flags** (category, summary, evidence)

A human flipping the status never deletes these records — an applicant can be `eligible / human` while still showing the AI flags a reviewer chose to accept. This preserves the audit trail.

**Stickiness:** a machine actor (rules or AI) must never overwrite a `human` status. On re-sync or re-run, machine actors refresh the reason/flag records but leave a human-set status untouched.

**Clearing an override:** a human override can be removed, handing the decision back to the machine. Clearing recomputes the status from the *current* findings (rules then AI) and resets `status_source` to the machine source — so the result can differ from the pre-override value if findings changed since (which is the point of reverting to automatic). The detail view models this as source ownership: a segmented **Decided by** control over `Automatic | Eligible | Ineligible`, where "Automatic" is selected whenever `status_source != human` and selecting it clears the override. The detail payload carries `autoStatus`/`autoStatusSource` (what the machine would decide right now) so the UI can show the automatic verdict even while a human owns the status. Clearing is idempotent — a no-op on an already-machine-owned status.

**Staleness nudge:** because human decisions are sticky, a re-run can surface new findings on an application a human already cleared. When the machine records change after a human's review, the application is marked stale ("new findings since last review") so the reviewer can re-decide. Status does not move; staleness is derived by comparing the latest machine-record timestamp to when the human set the status.

### AI Screening (Integrity Flags)

Separately from AI triage (which resolves ambiguous data), AI should make a screening/integrity pass over eligible applications to flag suspicious patterns that are too subjective or contextual for deterministic rules. When the AI pass flags an eligible application, it sets the status to `ineligible` with `status_source = ai` — the low-trust AI-excluded group — rather than excluding it outright. A human reviews these and either confirms the exclusion or restores the applicant to `eligible`. The flags themselves are kept as immutable records regardless of the human's decision.

The pass also re-analyzes applications a *previous AI pass* marked ineligible, not only currently-eligible ones, so that a revised prompt can change the verdict in either direction — clearing a previously-flagged applicant back to `eligible`, not just flagging clean ones. Applications the deterministic rules disqualified are excluded from the pass (rules outrank AI, so re-running AI on them cannot change their status). Human-set statuses remain sticky: their flags refresh for the staleness nudge, but the status is never overwritten by a machine run.

Known patterns to detect (this list is intentionally incomplete and should grow over time):

- Child names that look like placeholders ("Baby", "TBD", "N/A", "Test")
- Applicant or child names that appear fake or nonsensical
- Essay responses that are suspiciously short or minimal (single sentences, "N/A")
- Essay responses that appear to be advertising or spam
- Essay responses that appear to be AI-generated boilerplate with no personal detail
- Responses that are copy-pasted across multiple essay fields
- Internal inconsistencies between essays and other fields (e.g. claims skills they can't have given stated employment)
- Phone numbers or emails that appear fake beyond format validation (e.g. all same digits)
- Pet descriptions that violate the co-op pet policy (more than 1 dog, more than 1 cat, or exotic/unusual pets). The pets field is free text and too ambiguous for deterministic parsing — negation ("I don't have pets"), unclear phrasing, and context-dependent language require AI judgment.

AI screening flags should be stored per-application and shown in the candidate detail view as informational notices, not as filter reasons.

## AI-Assisted Screening

AI essay review should only run for candidates who pass deterministic hard filters or are resolved as eligible by AI triage.

### Provider And Cost Controls

For MVP, keep the AI architecture provider-adaptable. Because Jeff may work on this project from AWS-managed laptops where direct OpenAI, Anthropic, or similar external AI API calls may be blocked, Amazon Bedrock is the likely first provider to implement. Direct OpenAI or Anthropic providers can still be added later if useful, but the application should depend on an internal AI provider interface rather than a specific vendor SDK.

Jeff will set up AWS credentials, CLI, and SDK support on the relevant machines before Bedrock integration begins.

The app should run locally as much as possible for MVP while staying cloud-ready for eventual MOMI use. Local-first implementation should not prevent later deployment to AWS or another hosted environment.

Cost control is a core product requirement. The app should prefer:

- Cached AI analysis per application and per run where possible
- Smaller/cheaper models for first-pass extraction and clustering
- Frontier models only for higher-judgment synthesis, adjudication, and final report generation
- Batch/asynchronous processing where latency is not important
- Short structured outputs rather than verbose freeform generations
- Avoiding repeated spreadsheet reads and repeated AI calls
- A visible AI cost estimate before running large reviews
- A configurable spending cap per screening run

Hard filters should run automatically after import/sync. AI review should not auto-run by default; it should start only after the user sees the cost estimate and confirms the run, unless an Admin explicitly enables an auto-run option later.

### Interactive Screening

The screening experience should discover patterns in essay responses and ask the user what matters, rather than starting from a fully fixed rubric. The likely high-level criterion is "fit for Penta," but this is intentionally opinionated and user-dependent. The patterns are not pre-defined: the AI discovers the differentiating dimensions that actually distinguish *this* applicant pool (the "Known patterns to detect" list under AI Screening is for data-integrity flagging, not these screening differentiators). The screener is a screening assistant for a human, not an autonomous filter.

The assistant does not "cut" candidates. At the expected scale (~300 applicants) hard removal is the wrong model. Instead, the assistant **stack-ranks the entire qualified pool with a per-row rationale**, and the committee's weighting re-sorts that list. Re-weighting adjusts standing (soft ranking), never removes anyone — so a low-ranked candidate is never "rejected," just currently low, and a later weighting change can lift them back up. The committee reads the stack rank top-down as far as they like; there is no fixed cut line.

How the committee expresses what matters: a **tier-list maker** (built in milestone 9 — see "Interactive Weighting — Tier List"). The discovered dimensions are draggable chips the committee sorts into self-defined importance tiers (Critical/Important/Minor by default, plus an Ignore zone), and the ranking re-sorts instantly as deterministic math over the cached scores — no model call per change. This replaced the originally-envisioned sequential pairwise narrowing questions ("what matters more — X or Y?"): direct tiering states a committee's known priorities in one pass, and because every dimension stays visible and re-draggable, there is no "premature lock-in" for the questions' deliberate redundancy to guard against. (A future "Criteria Coach" may still *ask* questions — but to help the committee reflect on and challenge the weighting they built, not to elicit it; see the fast-follows in the M9 design section.)

Re-weighting is freely reversible: the committee drags chips back at any time, and the ranking re-sorts. There is no separate undo to maintain — the soft-ranking model makes revisiting automatic (re-sorting rather than un-rejecting).

Users do not need a manual "pin candidate" workflow for the initial design.

Rubrics should generally remain stable within a screening run. After a run completes, users should be able to revise criteria/rubrics for future runs.

### Ranking And Outputs

The primary output should be a ranked list. The ranking should be explainable and should preserve evidence behind each recommendation.

AI should produce qualitative labels for user-facing screening. Hidden internal scores may be used to support ranking, but the UI should explain rankings in plain language rather than centering numeric scores.

AI candidate summaries should use a neutral committee tone and avoid unnecessary personal judgments. They should still be transparent enough that users can detect bias, unsupported claims, or questionable reasoning.

AI judgments should show:

- Direct quotes or short excerpts where useful
- Paraphrased evidence
- Source field references
- Score or ranking breakdown
- Rationale for the recommendation
- Confidence or uncertainty labels where useful

Direct essay excerpts should be used sparingly as supporting evidence. The app should not reproduce entire essays in AI summaries or reports.

For debugging and learning, raw AI analysis, traces, prompts, and intermediate outputs are accessible to any logged-in member (see "Users, Roles, And Authentication" — the workflow has no admin-only surface). The normal app experience should emphasize polished summaries, with the raw detail in collapsible debug sections.

The app should provide `why not selected` explanations for candidates who do not make a member's shortlist, especially for merged MOMI comparison. These explanations are internal only; applicants will not see them, so they can be transparent and candid while remaining respectful and evidence-based.

Each screening run should be saved with its criteria, prompts, model outputs, user answers, ranking outputs, shortlist, and final decisions.

AI output schemas should be defined before implementing the AI milestone so prompts, storage, caching, UI rendering, and eval checks share the same contract.

### Essay Judgment

Strong negative essay signals include, but are not limited to:

- Applicant appears unaware of co-op obligations
- Applicant treats the unit mainly as cheap rent without understanding shared work
- Applicant expresses hostility or resistance to shared work
- Applicant has an unclear or inconsistent household situation

Essay concerns may justify a "do not interview" recommendation. Essay review is central to the screening process, not merely a low-priority flag.

Brief, awkward, translated, or non-native English essay answers should not be penalized for writing polish. The AI should judge evidence of co-op fit, participation commitment, and relevant signals rather than style, fluency, or grammar.

### Essay Analysis (Milestone 6) — REMOVED (see the M13 "essay-analysis PASS deleted" changelog note)

**This pass no longer exists.** It was deleted after measurement showed its digest inflated tokens ~172% over the raw essays while buying no discovery coverage; discovery and scoring now read the raw essays directly. The section below is retained as the historical design record of what M6 was. For the current pool view, see `pool_digest.py` and the "Pattern Discovery" section.

Milestone 6 adds a per-candidate essay-analysis pass on top of the shared AI foundation built in milestone 5 (the `analyze_application` engine, caching, cost estimate, spending cap, prompt versioning, narrative capture). It runs with `kind="essay_analysis"` through that engine, so it reuses all of the above for free.

The defining boundary: **milestone 6 extracts and normalizes what applicants said; it does not judge.** The "negative signals" and "do not interview" judgments described above under Essay Judgment belong to the milestone 7 ranker, because the differentiating criteria are *discovered* there against the actual pool. If milestone 6 emitted judgment on fixed dimensions, it would pre-commit the patterns and defeat that discovery. So essay analysis stays purely factual.

Decisions:

- **Output:** a neutral committee-facing summary plus structured per-signal fields. The summary lets a screener skim ~300 candidates without reading every essay; the structured fields feed the milestone 7 ranker.
- **Schema is fixed, not adhoc.** Normalization is the point — every candidate is described in the *same* structure so the committee can compare a column across the pool, milestone 7 reads a stable contract, and the output is evalable. The fields are locked; only the *contents* of list fields are open. The schema mirrors the four essay questions 1:1 — one field per thing the form explicitly asks for:

  ```python
  class EssayAnalysisReport(BaseModel):
      summary: str                          # 2-4 sentence neutral cross-cutting digest, no evaluation
      household_context: str | None         # who is in the household, as introduced (Q1)
      employment_background: str | None     # work situation as narrated, applicant + co-applicant (Q1)
      interests: list[str]                  # interests stated (Q1)
      values: list[str]                     # values expressed (Q1)
      skills_offered: list[str]             # skills offered to run/maintain the co-op (Q2)
      prior_co_op_experience: str | None    # prior co-op experience stated; null if none given (Q3)
      stated_motivations: list[str]         # reasons given for wanting co-op living (Q4)
      stated_contributions: list[str]       # ways they said they would be a valuable member (Q4)
      evidence: list[str]                   # short direct quotes grounding the extractions
  ```

  Extraction is cross-cutting (content bleeds across the four boxes — skills appear in the intro essay, etc.), not per-question. `null`/`[]` means "did not say," which milestone 7 may read as signal.
- **No `other`/catch-all field.** That would reintroduce the adhoc shape normalization exists to prevent. Off-question or cross-cutting nuance is covered two ways instead: the `summary` absorbs connective tissue, and the **raw essays and structured form fields are preserved**, so milestone 7's Pattern Finder reads the source directly and can discover anything the schema did not anticipate. Essay analysis is an *additive* lens, never a replacement for the source.
- **Model:** start on the first-pass model (Claude Haiku 4.5). Essays are short answers to fixed questions — extraction and summary, not the cross-document synthesis milestone 7 reserves for the synthesis model. Upgrade to Sonnet only if real output reads thin, decided empirically by comparing the same candidates. The model is Admin-configurable and the cost estimate self-tunes per model, so this is not a lock-in.
- **Scope:** eligible applications only. There is no value in summarizing essays for rules-disqualified applicants.
- **Status independence:** essay analysis is purely informational and never touches eligibility `status`/`status_source`. Unlike screening flags, it does not call `apply_machine_status`. Eligibility stays rules-and-flags driven; essay analysis informs the human ranking, not the gate.
- **Surfacing:** the summary and structured fields appear on the candidate detail page (committee-facing), but as a collapsed accordion among the other expandable sections near the bottom, *below* the raw essay responses — for a reviewer, reading the applicant's own words is the primary act, and the AI digest is a secondary aid available on demand.
- **No reasoning narrative.** Unlike the screening pass, essay analysis does *not* ask the model for a free-text reasoning preamble. An A/B run over real candidates (same model, with vs. without the "explain first" instruction) showed the preamble produces no systematic change in the extracted fields — the with/without difference sat within the model's own run-to-run nondeterminism — while costing ~18% more output tokens per candidate. Because this pass extracts short answers to fixed questions, the structured output needs no chain-of-thought scaffold, so the prompt returns the structured analysis directly. (The screening pass keeps its narrative: it makes a status-affecting judgment where the reasoning trail is worth the audit.)
- **Schema evolution is deliberate, never automatic.** If validation on real essays shows a signal that recurs across many candidates with no home and is too specific for the summary, add a *named, fixed* field — a human-approved, deliberate change (its prompt version re-derives automatically from the edited prompt text, re-running the pass). Never an `other` escape hatch, and never a runtime self-modifying schema (see the Screener-Evaluator role below).

### Pattern Discovery And Dimension Scoring (Milestone 7)

Milestone 7 is the AI foundation for ranking. It is read-only: it discovers *how this pool varies* and scores each candidate on those axes, but does not yet rank, weight interactively, or ask questions (milestones 8–9). It builds on the same `analyze_application`/`screen_applications` engine, caching, cost estimate, spending cap, and prompt versioning as milestones 5–6.

The defining architectural decision — **the LLM extracts scored features; ranking is deterministic math on top of them.** The model never produces "the ranking" *and never opines on importance.* It scores each candidate on a fixed set of discovered dimensions, and the ranking (milestone 8) is a plain weighted sum over those scores. Importance is a values judgment reserved for the human: the AI discovers *what varies*, the committee decides *what matters*. So milestone 8 starts every dimension at an **equal weight** — an honest "no judgment yet" baseline (fit is the plain average of a candidate's scores) — and milestone 9 is the only place weights diverge from equal, with every deviation traceable to a recorded human answer. A non-uniform AI-proposed default was rejected: it would quietly pre-commit the values question, presenting one applicant ahead of another before anyone said what they cared about. This is what makes the milestone 9 interactions — re-sort instantly as the committee re-tiers the criteria — cheap, instant, and *deterministic*: a weighting change only re-runs the math over cached scores, never re-invokes the model. Re-ranking the pool with the LLM on every answer was rejected: it is ~300× the cost per answer, slow, gives no cheap impact preview, and is nondeterministic (the order would jump for reasons unrelated to the answer). The SPEC's "hidden internal scores may be used to support ranking" points the same way — scores are the hidden support; labels and rationale are what the UI shows.

Two passes:

- **Pattern Finder (pool-level, one call, synthesis model).** Reads every eligible candidate's `EssayAnalysisReport` plus raw essays and discovers the **differentiating dimensions for this specific pool** — not a fixed rubric. Each dimension has a name, a definition, and a short why-it-differentiates note. It does *not* propose a weighting: importance is the committee's call, so weights are seeded equal at run creation and only the human moves them (milestone 9). Output is run-scoped (it describes the pool, not one candidate), stored on the `ScreeningRun`. Uses the synthesis model (Sonnet) because this is the cross-document judgment the synthesis tier is reserved for. This resolves the open question "pick a model for pattern discovery."
- **Dimension Scoring (per-candidate fan-out, first-pass model).** Scores each eligible candidate against the discovered dimensions: per dimension a score, a rationale, grounding evidence, and a confidence label. Starts on the first-pass model (Haiku) and upgrades to Sonnet only if an eval shows the scoring reads thin — the same empirical, measure-first stance milestone 6 takes on its model. The schema *shape* is fixed (`list[DimensionScore]`); only which dimensions appear is open, mirroring the `EssayAnalysisReport` discipline (fixed structure, open list contents).
  - **No call-level narrative (decided 2026-07-11).** Like essay analysis, the scoring prompt requests no free-text reasoning preamble (structured output only). So this pass's observability *is* its per-dimension `rationale` + `evidence`, surfaced on the candidate detail page — that's the per-decision "why did this applicant score 0.2 here?" trail. The call-level `narrative` field is therefore not persisted: one scoring call scores all N dimensions but splits into N cached rows, so storing the (unrequested, usually empty) preamble would duplicate one string across every row, read by nothing. This is the observability rule applied with judgment, not reflex: keep the reasoning that explains a decision (the rationale), drop the redundant preamble that doesn't.
  - **Two Rank modes.** When a current criteria set exists, the workflow offers a safe default alongside full discovery: **Score missing applicants** runs this existing scoring pass only for eligible applicants missing at least one current-dimension result. It preserves the `RankingRun`, its dimensions, and the committee's tier layout; its estimate is calculated from the current cache misses and it is independently cap-gated. Complete score coverage makes the retained criteria current for the workflow: choosing this mode is the committee explicitly accepting that criteria set for the changed pool. **Discover new criteria** remains the full, nondeterministic discovery → decomposition → matching → scoring chain and may replace the criteria set. Both modes record their fresh scoring cost in Insights.

Design constraints carried from the rest of the SPEC:

- **Scope:** eligible applications only, like essay analysis. Status-independent — scoring never touches `status`/`status_source`; it informs ranking, not the gate.
- **Cache key must include the dimension identity.** The shared cache key is `(raw_row_hash, kind, model, prompt_version)` and does *not* see the prompt body directly — but `prompt_version` is now **derived by hashing each pass's static prompt text** (system prompt + instruction template + any shared fragment it embeds), so a prompt edit changes the version and invalidates that pass's cache automatically (see "Prompt versioning is derived" below). A candidate's score also depends on *which* dimension it is. Scores are cached **per dimension**, with the dimension's **key** folded into the `kind` (`dimension_scoring:<dimension_key>`), so each (candidate, dimension) gets its own entry and a dimension that recurs under the same key across runs reuses its score. (Originally a whole-set `dimension_scoring:<dims-hash>` keyed one row per candidate; M9 carry-forward Phase 4 moved it to per-dimension keys for cross-run reuse — see "Tier Carry-Forward On Re-Rank".)
- **Run-scoped, building on the existing `ScreeningRun` table.** The discovered dimensions are a property of a run, persisted in `ScreeningRun.criteria` (the table and JSON column already exist but are currently unwired). Milestone 7 wires the minimum the foundation needs; milestones 8–9 accrete weights, answers, and rankings onto the same run rather than adding a separate persistence milestone. Milestone 8 seeds `criteria.weights` (one entry per dimension key, all equal); the ranking engine reads `criteria.weights`, never a per-dimension field, so this map is the single seam milestone 9's tier-list mutates.
- **Surfacing (read-only):** the run's discovered dimensions are shown at the screening level; each candidate's per-dimension scores, rationale, and evidence appear on the candidate detail page. No ranked order yet — that is milestone 8. Numeric scores stay supporting detail; the committee-facing emphasis remains qualitative, per "Ranking And Outputs."
- **Schemas defined first.** `PoolPatternReport` (dimensions only — no weights; importance is the human's call) and `DimensionScoringReport` (`list[DimensionScore]`) land in `app/ai/schemas.py` before prompts/UI, per "AI output schemas should be defined before implementing the AI milestone." This resolves the open question "define schemas for pattern discovery and ranking."
- **Inputs: essays *and* structured facts.** Both passes see a shared `applicant_facts` view (household composition, ages, income + applicant/co-applicant split, employment tenure, pets) alongside the essays, so discovery can surface quantitative axes (income mix, employment stability, household-to-unit fit) and scoring can score them from the same facts. The view is defined once (`app/ai/applicant_facts.py`) so the two passes never drift — a fact-based dimension must be scoreable from the identical fact. Excluded: names/emails/phones (identifiers, no screening value) and **real-estate ownership** (a hard filter, so eligible applicants are uniformly non-owners — no residual signal). Fields that *are* hard filters but still vary (income within band, household size, pets) are framed for **residual variation** only (`FILTERED_FACTS_NOTE`): the model reads the variation that remains among the already-qualified pool, never the constant pass/fail fact, and only treats applicant-provided context as meaningful when it is relevant to the co-op criteria being analyzed.
- **Dimension count is a guided range, not a fixed number.** Discovery targets **5–25 dimensions**, biased to *split* a broad axis into distinct, separately-weighable sub-dimensions rather than merge, but told explicitly not to pad to a number or invent axes the data does not distinguish. The range is wide because the right count depends on how richly the pool actually varies; the anti-padding guardrail keeps the ceiling from acting as a target. (Empirically, the real pool yields ~14–16.)
- **Dimensions are oriented so MORE is better fit (decided 2026-06-28).** A 0..1 score always pushes fit up (`fit = Σ weight·score`), so scoring can't be neutral — direction is baked into the dimension at discovery, not left implicit. The prompt makes every dimension's high end the desirable end (recasting "less is better" axes to their positive form, e.g. "frequency of breakdowns" → "mechanical reliability"). For **"goldilocks" axes** (best value in the middle) two paths: (a) a peak from ONE quantity vs. a target → *reframe* to the underlying monotonic fit-concept ("amount of salt" → "seasoned about right") — confirmed working (income level → distribution balance); (b) a peak from TWO opposing forces a household could have one without the other → *emit two* separate more-is-better dimensions and let the committee's weighting place the peak, never a merged "balanced X" — this is the same OPPOSING-EVIDENCE split the one-concept-per-dimension rule already uses, and is confirmed working on a real run (committee request "strong primary earner, but not single-income-dependent" → split into `primary_earner_income_strength` + secondary-income-contribution). The committee owns direction only by Ignoring an axis or proposing a rephrase; there is **no** per-dimension direction flag in the schema (an earlier `more/less/undecided` enum + sign-aware ranking was designed and reverted — orienting at discovery is simpler and needs no ranking-math change). Examples in the prompt are kept out-of-domain so they teach the move without leading the pool-specific answer.
- **When the two-dimension split actually fires (empirical, ~8 real runs on the synthetic pool).** The split (path b above) reliably fires only when the two forces are **independently measured** — distinct quantities where a household can be high on one and any value on the other (the income split: primary-earner strength vs. secondary contribution, two separate numbers). For **single-variable, two-reading axes** — one fact read in opposite directions, where one pole is essentially "less of" the other — the model does **not** split no matter how strongly the prompt pushes or how loud the pool signal: it either drops the axis or, better, **absorbs it as a sub-signal of a measured axis it is already scoring**. Two cases proved this repeatedly: *child age* (toddlers↔teens) — across runs the model smuggled a direction, dropped it, or (cleanest) folded youngest-child age into `long_term_residency_intent` as a tenure-horizon input; and a deliberately loud essay-seeded *civic-engagement-vs-spare-capacity* tension — the model folded availability into `participation_commitment` (low spare hours read as a weak/"carve-out" participation score) rather than emitting a separate capacity axis. **This is accepted behavior, not a bug to prompt around:** pushing harder produced fusion artifacts (a slash-named "Age Profile / Adult Bandwidth" dimension), never a clean split. The absorb-or-decline instinct is the one-concept discipline working correctly; a soft single-variable trait genuinely *is* better scored as part of a nearby measured axis. (Child age also keeps self-flagging as family-composition the model is reluctant to rank on directly — the parked descriptive/"committee-decides" gap; revisit there, not via the split.)
- **Discovery is uncached and not cap-gated, but the whole Rank chain is gated on a pool fingerprint.** Pattern discovery is a single synthesis call that writes to `ScreeningRun`, not through the per-application cache — so it is uncached and has no derived `PROMPT_VERSION` at all (only the cached passes do). It is not cap-gated (one cheap call, ~$0.07–0.12). Its model call is wrapped so a Bedrock failure returns a readable 502, not a bare 500. Scoring *is* cap-gated and cached; because re-discovery changes the dimensions-hash (and thus the scoring `kind`), it forces a re-score. **That last property is exactly why a no-op re-run is wasteful:** discovery is nondeterministic, so re-running on an *unchanged* pool churns the dimension wording → new dims-hash → a full, pointless re-score (~$0.40). So the Rank chain (essays → criteria → scores) is gated on a **pool fingerprint** — a hash of the sorted `raw_row_hash`es of the eligible pool — stored on the run when it completes. If the fingerprint is unchanged since the current run, re-ranking is **blocked** (`/rank/run` → 409; `/rank/estimate` returns `ranking_current: true` so the UI says "ranking is up to date" instead of offering to spend). The pool must actually change — a new applicant, an edited application, or an eligibility flip — to re-rank. This supersedes the earlier "re-running always produces a fresh run" behavior: the committee can only re-rank when there is new information to rank on, which is the same per-essay→criteria→scores implication chain the dependency already encodes (no new essays ⇒ no new criteria ⇒ no new scoring; even one changed application re-opens the whole chain, since discovery reasons over the whole pool).
- **Surfacing UX (built in M7, simplified post-M8).** The screening workflow is an ordered, gated strip in its own full-width row, named with single verbs: **Import → Screen → Rank**. Import is sync (which runs the deterministic hard filters). Screen is the AI screening/integrity pass (flag suspicious submissions). **Rank is one button that runs the whole essays → criteria → scores chain** — the committee never runs those sub-passes individually, so they are collapsed into a single step under one combined cost estimate, with the cap enforced once over the combined cost. The three passes stay separate underneath (distinct schemas, cache kinds, and status behavior); only the UI and the orchestration endpoint are merged. The standalone per-pass endpoints were removed when the button merged (`remove completely` per the engineering rules); `screen_essays`, `discover_patterns`, and `score_dimensions` are still the underlying passes, called by `POST /screening/rank/run`. Each AI step shows live coverage of the *current* eligible set (`cached/inScope`). A step goes **amber-stale** when it has run but is no longer current. Crucially, "current" is defined per step by the *same* signal its no-op gate uses, so the badge and the run button never disagree: **Import** stales on a **settings fingerprint** change (`importCurrent: false` — the import-relevant settings, i.e. sheet id + hard-filter thresholds + disabled rules, changed since the last sync, so a re-import would reclassify eligibility; pet limits and the AI cap are excluded since they don't affect import); **Screen** stales on coverage falling short (`cached < inScope`, i.e. a re-sync added uncovered candidates); **Rank** stales on the **pool fingerprint** changing (`rankingCurrent: false`) — not on coverage. (Import's green is "probably fresh," not a guarantee — we can't detect an edited spreadsheet — but a settings change is near-certain grounds to re-import.) This distinction matters because a pool change can leave Rank's score coverage *full* (e.g. toggling a previously-scored candidate back into the eligible pool): coverage alone would wrongly read "done/green," but the fingerprint correctly flags a re-rank is warranted. The dashboard returns `rankingCurrent` (computed by the same `ranking_is_current` the gate uses) precisely so the workflow strip reflects gate truth. Rank streams phase-aware progress (essays / criteria / scores). The essays and scores phases process N candidates, so they show a real `processed/total` bar; the **criteria phase is one multi-minute model call** (discovery + the carry-forward match) with no per-item progress, so instead of a bare spinner it **streams the model's live reasoning** as an auto-scrolling, markdown-rendered "thinking" panel (with a "this can take up to 5 minutes" caption as the floor). This rides the provider's single `structured_output(..., on_delta=...)` method: there is **one** call path — it always drains Strands `stream_async` (a spike confirmed that's safe at the worker pool's ~50-wide fan-out, each thread getting its own loop via `asyncio.run`), and `on_delta`, when passed, fires per reasoning chunk; callers that don't want deltas simply omit it. (An earlier two-method split — sync vs. streaming — was collapsed: keeping both doubled the interface and forced an `if on_delta` branch at every call site, for no behavioural gain.) The result `AIResult` is identical with or without a sink. `rank_run` runs the criteria passes on a worker thread that pushes deltas onto a queue, and the NDJSON generator drains them as `criteria_thinking` events (a generator can't yield from a callback). Only discovery/match pass a sink — the per-candidate passes omit it and keep their honest `processed/total` fractions. **A completed Rank lands the user directly in the ranked view** (the ranking is the point of the run; the separate "View ranking" button was easy to miss). **Every AI step opens a confirmation card before running — always, even when there's nothing to do.** When a step is already up to date (Screen: nothing uncached; Rank: pool unchanged), the card states that and offers only **Close** (no run button — the action would be a server-blocked no-op), rather than firing a transient toast the user might miss. Run *completions* and *errors* still surface as toasts (green auto-dismiss; red persistent and copyable).

### Deterministic Ranked List (Milestone 8)

Milestone 8 turns the M7 per-candidate scores into the ranked shortlist. **No new model calls** — ranking is pure deterministic math over the cached `DimensionScore`s, which is exactly what makes the M9 interactions instant, free, and reproducible. Design decisions:

- **Equal-weight baseline.** Weights are seeded uniform at run creation (`criteria.weights`, one entry per dimension key). Fit for a candidate is the weight-normalized average of their per-dimension scores: `Σ(weight·score) / Σ(weight)` over dimensions with weight > 0. At M8 this is a plain average; M9's tier-list is the only thing that moves weights off equal. The AI never proposes importance (see the "Pattern Finder" note above) — discovering *what varies* is the model's job; deciding *what matters* is the committee's.
- **Confidence is surfaced, not discounted.** Each `DimensionScore` carries a confidence label; it is shown next to the score and rationale but **never folded into the fit number**. A score moves the ranking by exactly its weight and nothing else, so the ranking stays explainable top-down. (Confidence-weighting was considered and rejected for M8: it adds a hidden term that makes "why is this candidate here" harder to answer.)
- **Qualitative labels are relative bands, not fixed thresholds.** The committee-facing label on each row ("Strong fit", "Promising", "Mixed", "Limited") is assigned by the candidate's *position within this pool* (rank percentile), not an absolute score cutoff. This matches the "how does THIS pool vary" framing — bands always spread the pool and recompute as weights change — and keeps numbers as supporting detail, per "Ranking And Outputs." The raw fit number is available but never the headline.
- **Ranking is a pure domain function.** It lives in `app/domain/ranking.py` alongside `hard_filters.py` (deterministic logic, separate from AI per the engineering rules), takes already-fetched scores + weights and returns ordered rows with fit, band, rank, and per-dimension contributions preserved for explainability. No DB or provider access in the function itself — trivially unit-testable with hand-built scores, no mock provider needed.
- **No fixed shortlist line.** An earlier M8 build had a configurable "shortlist line" with a live above-line count; it was removed as unhelpful — the list is stack-ranked, so the committee simply reads top-down as far as they like and draws their own line by eye. (Removed completely: `criteria.shortlist_size`, the `/screening/shortlist-line` endpoint, and the `above_line` flag.)
- **Surfacing: a separate ranked view, not an in-place re-sort** of the eligible table. Ranking is run-scoped and grows the M9 tier-list beside it, so it gets its own view rather than re-sorting the browse-applications table. The workflow strip is unchanged (ranking is a view over completed scoring, not a gated AI step). `GET /screening/ranking` returns the ordered rows + current weights; it 409s if patterns/scores don't exist yet, matching the scoring endpoints.

### Interactive Weighting — Tier List (Milestone 9)

The M8 equal-weight ranking was validated against the real pool and judged not good enough — every discovered dimension counting equally does not reflect what the committee values. M9 lets the committee say what matters and re-sorts instantly, as **deterministic math over the cached `DimensionScore`s — no model call**.

**The interface is a tier-list maker, not sequential pairwise questions.** The committee drags the discovered dimensions into **importance tiers they define themselves** — from 2 tiers (Important / Ignore) to one-per-dimension (a strict stack rank), most landing in between — plus a bottom **Ignore** zone (weight 0). This was a deliberate move away from the SPEC's original "what matters more — X or Y?" framing:

- **Direct beats indirect for a committee that already has opinions.** Backing into a known preference via many pairwise questions is slow; dragging dimensions into tiers states it directly in one pass, and can zero out whole groups at once ("only the top 3 matter").
- **Always-editable controls remove the lock-in that pairwise redundancy guarded against.** The original design wove in redundant/overlapping questions so no single early answer "locked" the order prematurely. With a tier-list every dimension is visible and re-draggable at any time, so there is nothing to lock — the anti-lock-in machinery (and the constraint-solver that would integrate overlapping comparisons) is unnecessary.

Design:

- **Tier layout is the source of truth; weights are derived.** The run stores `criteria.tiers` (ordered list of `{id, label, dimension_keys}`, most→least important, with a conventional Ignore tier). A pure `weights_from_tiers(dimension_keys, tiers)` recomputes `criteria.weights` from the layout: non-ignore tiers get a descending weight by position (top = N, …, 1; equal within a tier), Ignore = 0. Every weight traces to a tier position — maximally auditable, matching "explain rankings in plain language / traceable to a recorded human choice." The exact weight curve is a tunable constant.
- **The ranking engine is untouched.** `weights_from_tiers` writes the same `criteria.weights` map `rank_candidates` already reads (M8). Re-sorting on a tier edit is the existing pure math; M9 adds only the tier→weights derivation and its persistence.
- **Default layout = Critical / Important / Minor + Ignore, with every dimension starting in Important.** The named tiers give the committee an immediate tier-list mental model and room to both promote (to Critical) and demote (to Minor / Ignore); starting everything in one tier keeps the weights uniform, so the opening ranking is still the M8 equal-weight baseline (weight-normalized fit makes "all in A" identical to "all equal") until the committee moves something. The labels are just the default — tiers can be renamed, added, and removed freely.
- **Deterministic and trivially reversible.** Same tier layout → same weights → same order. Undo/redo is editing the layout; there is no separate weight state to keep in sync.
- **No AI, no cost gate in v1.** Manual tiering only — a tier-list UI (`@dnd-kit` for accessible drag), a pure derivation function, and thin persistence (`GET`/`PUT /screening/tiers`; PUT validates keys against the run's dimensions, persists `tiers` + derived `weights`, and returns the fresh ranking). Both 409 before a run exists.
- **Run-scoped (v1 behavior; superseded by carry-forward below).** Tiers reference dimension keys, so they belong to a run. In v1, re-running Rank makes a new run with fresh dimensions and a default layout, discarding the old tiers — a known sharp edge. Fast-follow #1 ("Tier Carry-Forward On Re-Rank", below) fixes this by matching new dimensions to old ones and carrying placements forward.

**Fast-follows (in order), recorded so they aren't lost:**

1. **Tier carry-forward across a re-rank** (next — designed in detail below under "Tier Carry-Forward On Re-Rank"). Today re-ranking discards the committee's entire tier setup; this is the fix.
2. **Add a dimension mid-tiering** (after carry-forward). Dragging dimensions around is exactly when the committee notices a factor the AI missed, so the tier-list grows an "add dimension" affordance. A user-added dimension gets a **new key**, so only it is scored for each candidate (every existing dimension's score is reused under its key), then it appears as a chip. No new AI surface — it reuses the per-dimension scoring pass. This relocates the parked "human-editable criteria" enhancement into the M9 flow where the need actually arises. (The per-dimension scoring cache that powers this is now built — Phase 4 above — so adding one axis costs one dimension's worth of scoring, not a full re-score.)
3. **Surface weak spots, not just top drivers** (no AI — pure presentation). **Done — solved at the root in two layers.** Two distinct defects hid weaknesses:
   - **A label bug (fixed first).** Each contribution line printed the dimension's **confidence**, not its **score**, coloured on the confidence ramp (high→green). So a candidate with *no* co-op experience showed "Residential Co-op Experience (High)" in green — a confidently-identified *weakness* reading as a strength. Fixed by showing the **score band** (`Strong ≥0.66 / Moderate ≥0.33 / Weak <0.33`, via `scoreBand()` in `App.tsx`) coloured by merit (strong→green, moderate→blue, weak→amber). The colour now tracks merit, and the (deliberately neutral-grey) confidence sits beside it without implying merit it doesn't carry.
   - **A selection bug (the deeper fix).** The row chose its few dimensions by `weight × score`, which structurally selects *strengths* — a heavy dimension a candidate scores *low* on (the biggest strike a committee should see) could never make the cut, because a middling strength always out-scores it on `weight × score`. The fix is the right metric: **`impact = weight × (score − pool_mean)`** — the exact per-dimension decomposition of `fit_i − avg_fit` (the shared `/ΣW` normalizer drops out, so impacts are comparable within a candidate). Ranking contributions by `abs(impact)` surfaces what *moved* this candidate up or down: a dimension everyone scores alike has ~0 impact (it changed nobody's rank, so it's correctly demoted however heavy); a low score on a heavy dimension has large *negative* impact and leads the row as the amber strike it is. **Sign carries direction; magnitude carries importance; the score band's colour says strength-vs-weakness.** So a row now reads both why a candidate ranks high *and* where they're weak — exactly "prioritise weaknesses as well as strengths."
   `impact` is computed in the pure ranking domain (`app/domain/ranking.py`, with pool means averaged over candidates that have each dimension scored) and rides on every `DimensionContribution`. **The candidate detail page is now the same contribution objects** (a single shared assembler, `app/services/ranking_view.py`, builds the ranker input for both the shortlist and the detail page, so the `impact` formula lives in exactly one place), **ordered by `abs(impact)` to match** — the ranked-list row is literally the top slice of the detail page's list, so the two views tell one story. The detail page colours its score bar and a word-band by merit and keeps confidence neutral, and **drops weight-0 (Ignored) dimensions entirely** — they contribute 0 to fit and 0 impact, so they are irrelevant to the ranking and would only clutter the page with axes the committee chose not to weigh. Surfaced from the end-to-end test that first read as "no low scores in the list" — the lows were always there (44 low-confidence, 115 score<0.34 of 377); the row was both mislabelling them and selecting them away.
4. **AI Criteria Coach** (later, after hands-on use). *Not* a propose-the-tiering tool — its role is to **help the committee understand the weighting they built and challenge it**: surface tensions ("income mix is in Ignore, but three of your top candidates lead mainly on it — intended?"), flag near-ties, prompt reflection. Deliberately deferred until the tier-list has been used against real data, because what is worth challenging only becomes clear from use. A cheap synthesis call when it lands.

#### Tier Carry-Forward On Re-Rank (M9 fast-follow #1 — building, phased)

**Build status:** Phases 1–5 complete. Phases 1–3 + two hardening passes were **verified end-to-end against real Bedrock** (ran the full Import→Screen→Rank chain, tiered dimensions, forced a pool change, re-ranked: 6/8 placements carried across drifted keys, an ambiguous match correctly declined, unmatched dims parked in Ignore and badged, badge cleared on triage). Phases 4–5 (per-dimension score reuse + ceiling estimate) are built and unit-tested; a real-Bedrock re-rank to confirm the cost win in the wild is the remaining validation.
- *Phase 1 (foundation):* `weights_from_tiers` falls back to uniform when *no dimension has positive weight* (not just when there are zero non-ignore tiers); `default_tier_layout` starts every dimension in **Ignore** (Critical/Important/Minor empty). (A "nothing weighted yet" baseline label was considered and dropped.)
- *Phase 2 (match pass + carry-forward):* `app/ai/dimension_matching.py` is the second pass — given the prior and new dimension lists (name + definition), the first-pass model returns high-bar, one-to-one `{new_key → old_key}` identity matches (`DimensionMatchReport`); the result is sanitized (unknown/duplicate keys dropped). `carry_forward_layout` (in `screening_run.py`) clones the prior run's tier *structure* and re-places each new dimension into its matched tier, sending unmatched ones to Ignore and returning them as `new_dimension_keys`. `create_run` now persists `tiers` (carried or default) + `new_dimension_keys` and derives `weights` from the layout. The re-rank chain runs blind discovery → match → carry-forward; `criteria_done` reports `carriedForward`/`newDimensions`, and `GET /screening/current` exposes `newDimensionKeys`. First run skips the match pass (no prior report) and opens with empty working tiers, nothing flagged. **Ignore modeling (refactored):** rather than store Ignore as a tier with a must-exist invariant, the run stores only working tiers and treats "ignored" as the absence of a placement (`weights_from_tiers`: unplaced → weight 0). `display_tiers` synthesizes the Ignore zone for the UI; `set_tiers` strips it on save; `stored_tiers` is a plain read of working tiers. (The DB was reset rather than carry a shim for the old explicit-Ignore shape.) See "Ignore is the absence of a placement" above for the rationale.

- *Phase 3 (new-dimension badging):* the tier-list reads `newDimensionKeys` and badges each unmatched-new dimension with an amber "New" marker **while it sits in Ignore**. Threaded `newKeys` through `TierList → TierRow → DimensionChip → ChipBody`. First run flags nothing.
- *Bug fix (post-Phase-3):* `carry_forward_layout` had branched on whether a match landed in a *working tier*, which wrongly badged dimensions that matched a prior **Ignored** dimension (their match-key isn't in the working-tier map, since Ignore isn't stored) — e.g. `long_term_residency_intent`→`long_term_stability_intent`, even byte-identical keys. Fixed to branch on `old_key is None` (no match at all). "New" now means *unmatched*, not *unplaced*. (Found via a debug query on the test run — all Ignore dims were flagged.)
- *Acknowledge in place:* `new_dimension_keys` reframed as "**unacknowledged** new dimensions." A member clears a badge two ways, both folded into the tiers PUT (`acknowledged_keys` field) and recomputed in `set_tiers` as *prior-new − acknowledged − placed*: drag into a working tier, **or** acknowledge in place via the per-badge ✕ or the "Clear all N new flags" action on the Ignore row. The PUT response returns the recomputed `newDimensionKeys` so badges refresh in one round-trip; the tier-list reads them from `ranking` (live), not the static `screeningRun`. Also fixes a wrinkle: dragging a new chip to a tier and back to Ignore no longer re-shows the badge.

- *Phase 4 (per-dimension score reuse) — DONE.* Scores are cached at **per-(candidate, dimension)** granularity, keyed by the dimension **key**, so a dimension that carries forward across a re-rank **reuses its cached score** instead of re-scoring the whole pool. The key insight (a deliberate simplification over an earlier opaque-"lineage-id" design): the cache and the tiers can share *one* identity — the key — if matched dimensions simply **adopt the prior key**. The pieces:
  - *Key adoption (`screening_run.adopt_matched_keys`).* After the match pass, every matched new dimension has its `key` rewritten to its prior counterpart's key, **keeping the new name/definition/why** (discovery just re-read this pool, so its wording is the current, accurate one — only the opaque identifier is borrowed). Then a matched dimension *is* the prior dimension by key: its tier placement and its cached score both carry forward by key alone, with no parallel id to maintain. A guard prevents creating a duplicate key (the rare case the LLM re-coins an old key for a different unmatched concept keeps its own key).
  - *Cache shape.* Scoring rows are **one-per-(candidate, dimension)** under `dimension_scoring:<dimension_key>`, each storing a single `DimensionScore`. (`dims_hash`/`dimensions_hash()` from the original whole-set design are deleted.) Cache identity is the key, **not** the definition text — the match pass vouches a carried-forward dimension is the same intent; a *future* "edit a definition" feature would change the key to force a re-score.
  - *Batched scoring (`score_dimensions`).* Per candidate, the uncached dimensions are scored in **one** call (never one-per-dimension — facts + essays are never repeated), the call's tokens are split evenly across the dimensions it scored when the per-dimension rows are stored, and cached + fresh scores are merged into the candidate's full report. A fully-cached candidate makes no call.
  - *Shared concurrency core.* The thread-pool / session-on-main-thread / error-isolation mechanism was extracted from `screen_applications` into `run_in_pool`; `screen_applications` stays the 1:1 wrapper for screening flags + essays (behavior unchanged), and scoring uses the same core without bending the 1:1 abstraction. (The subtle session-safety discipline now lives in one place.)
  - *Assembly + coverage.* `ranking_view.candidate_scores(db, run)` reads each candidate's per-key rows and merges; dashboard `candidatesScored` counts a candidate as scored only when it has a row for *every* dimension key (partial coverage reads as incomplete). `carry_forward_layout` now matches by key equality against the prior key set (no match map needed — adoption already happened).
- *Phase 5 (estimator + confirmation) — DONE.* The pre-run Rank estimate prices scoring as a **whole-pool ceiling** (every candidate × every dimension): the estimate is shown before discovery and the match pass run, so it cannot know how many dimensions will carry forward — and discovery is nondeterministic, so guessing savings would risk an under-count that erodes the cap guarantee. The per-dimension cache makes the *actual* run come in under the ceiling; the real saving shows in the run summary. The match-pass cost (`dimension_matching.estimate_match`, one small first-pass-model call) is folded into the combined estimate and the single cap check, but only when a prior run exists (otherwise the match pass is skipped). The confirmation card shows the match line (re-ranks only) and notes scoring is an upper bound that carry-forward lowers. Dimension count for the ceiling comes from the current run, or a constant on a first run.

Carry-forward is now complete end to end (Phases 1–5).

**Matching scope: all-history, not last-run (decided 2026-07-08).** The match pass and tier carry-forward originally looked at **only the most recent run**. That caused a real, frequently-observed bug: a concept that fell out of a run and re-surfaced was matched against a prior run that no longer contained it → the model minted a **new key** for it → the distinct-key count grew ~4/run and never converged (17 runs had produced **67** distinct keys for a pool that varies on only ~20–25 concepts), *and* — because the score cache is keyed by dimension key — the resurrected concept either got a fresh key (needless re-score) or, worse, collided with an abandoned key string and silently read a **stale score from two runs ago**. The "accepted corner" documented above was in fact this bug, hit often.
- **Fix (A):** `all_known_dimensions(db)` builds a synthetic report of **every dimension ever discovered, one entry per key (latest definition)**, and the match pass matches the fresh discovery against *that*. A re-surfaced concept now re-adopts its existing key instead of minting a new one, so the key count converges on the true concept count and cached scores are legitimately reused. Cost is negligible — one match call, ~50 tokens/dimension; even ~70 dims is a few thousand Haiku tokens.
- **Tier carry-forward widened to match.** `tier_history(db)` returns the **most-recent tier placement of each key across all runs** + the set of **all keys ever seen**; `carry_forward_layout` restores each adopted key to the last tier the committee placed it in, from whichever run that was. **"New" now means never-seen-in-any-run**, not "absent from the last run" — a placement is durable committee intent that survives a dimension dropping out and re-surfacing (a key last left in *Ignore* returns to Ignore, still not badged new). This keeps score reuse and tier restoration consistent: adopting a key restores *both*, per the one-identity principle. Discovery **proposals** stay last-run — they're the committee's *pending* asks, not durable history. (Favourites, an earlier second seed type, are retired — automatic reconcile now keeps prior axes alive; see "Committee-Proposed Criteria.")
- **Chosen over (B) "scope the score cache to the match verdict."** B (only reuse a score if the key was carried forward *this* run) also fixes the stale-score corner, but A additionally fixes the root cause — runaway key growth — and *improves* caching (more legitimate reuse), which was the decisive win. **Potential future change:** if widening the match list is ever seen to raise *over-matching* (a wrong identity match reused across a bigger candidate list — now costly since it reuses a score, not just a tier chip), revisit adding B as a backstop, and/or a per-run cap on match-list size. The `match_audit` carry-forward-rate view is how we watch for this.

**Post-score consolidation — the Pearson-nominate / LLM-confirm cleanup (BUILT + verified live 2026-07-11; matching prevents, consolidation heals).** *Verified on the real pool: the first fresh Rank nominated 6 correlated pairs and merged 1 — `financial_stewardship_skills → professional_financial_management` (r=0.885), healing the exact run-16 fork — while correctly KEEPING 5 confounds apart, including a 0.94 pair (`financial_stewardship_skills ↔ financial_literacy`: reading/critiquing finances vs. doing stewardship) the model held apart on definitions. 1 real merge, 5 correct keeps, 0 over-merges — the merge-by-default-but-reject-confounds calibration landed. Alias written; the fork is healed for future runs via `all_known_dimensions` resolving through it. Surfaced in Insights (Consolidation tab) + cost on both ledger surfaces (drift-guarded).*** Matching's job is *cost avoidance* — adopt a prior key so scoring reuses cache — and it works, but it decides on *definitions* only (fresh dimensions aren't scored yet), with a deliberately high bar. So it occasionally misses a re-worded concept: one run mints a near-duplicate key next to an established one, and because both are then established priors, no later match dares merge them → the dupe is **born in one run and preserved forever**. Traced on real data: `financial_stewardship_skills` was minted in run 16 alongside `professional_financial_management`; both carry forward faithfully thereafter (r=0.94 on score vectors — same axis). The distinct-key count grows while the true concept count is ~20-25. **The fix runs *after* scoring, where matching can't help but a new signal exists:** every dimension in the run now has a score vector, so a genuine duplicate reveals itself as near-identical vectors. Design:
- **Placement: a new pass after scoring, before the run's cost is finalized.** Post-score is deliberate — this run's dimensions now have score vectors. Runs on the criteria worker like the other passes.
- **Scope: ALL dimensions, nominated as current-run × all-known-history — not new-only.** New-only would structurally miss the target: the run-16 fork only *coexists* from run 17, when BOTH keys are established priors (neither "new"). So nominate any pair where ≥1 key is in this run, comparing against every key's cached vectors. This heals the fork in any run where *either* variant surfaces (a this-run key correlates against a prior key's cached vector even when the prior didn't surface this run), which **guarantees convergence** rather than waiting for both to coincide in one run. Cost is identical — Pearson is free; only flagged pairs hit the LLM.
- **Pearson NOMINATES, an LLM CONFIRMS — not the coefficient alone.** Correlation is a strong net but a bad verdict: on real data, within-run pairs at r ≥ 0.85 were 2 genuine duplicates and **3 confounds** (`community_motivation_alignment ↔ task_completion_followthrough_reliability` at 0.854 — "why you're here" vs. "do you finish things"; engaged people score high on both). A pure-threshold auto-merge would over-merge 3-of-5 — and over-merge is the *invisible, unrecoverable* error (a lost distinction never resurfaces; the whole decompose design guards it). So: correlation flags candidate pairs deterministically at the default r ≥ 0.8, then **one cheap LLM call adjudicates the flagged pairs by their definitions** — merge only on confirmed-same-concept. The judge sees the definitions (ground truth) plus "these score applicants near-identically."
- **Merge = alias the losing key to the winner (the older/prior key wins)** to maximize cache carry-forward — the newborn is aliased to the established identity, exactly like match adoption (scores + tier placement follow the surviving key). **The alias is durable and feeds the match pass**: `all_known_dimensions` resolves through it, so a future run's re-minted duplicate is adopted onto the canonical key at match time — otherwise discovery re-mints it and we re-heal every run. Sticky: a confirmed merge stays merged; a borderline r that wobbles below threshold later does not un-merge.
- **First-class observability + costing (per the AI-call rules).** The confirm call is a judgment, so it follows every other AI pass: **persist its reasoning** (which pairs, merge/keep + why — the confabulation/over-merge audit trail), track its **cost** in the run ledger as its own pass ("Dimension consolidation"), **estimate** it in the pre-run Rank projection (history-first like the others; ~0 most runs), and surface it in **Insights** (pass row + reasoning panel, alongside discovery/decompose/match). A `consolidate_audit` on the run mirrors `decompose_audit`/`match_audit`: nominated pairs, the verdict per pair, the alias applied.
- **Why this is the right division of labor (Jeff's framing):** match = pre-score, definition-based, *prevents* unnecessary scoring within a run (worth its cost); consolidation = post-score, vector-nominated + definition-confirmed, *cleans up* the duplicates matching's definition-only view couldn't catch. Together the key space finally **converges** instead of growing — dimensions collapse toward the true ~20-25 concepts.

**Consolidation stays definition-based identity merge — it does NOT auto-deactivate on correlation (decided 2026-07-12).** Considered and rejected: having consolidation *deactivate* one of any highly-correlated pair (not just definitional duplicates) to prevent double-counting. The rejection rests on separating two different problems that "double-counting" conflates:
- **Identity duplication** — two axes that *are* the same concept (`pet_ownership`/`pet_load`). Merge is correct and *safe*, because the LLM confirms by **definition** before collapsing. This is what consolidation does.
- **Statistical redundancy** — two *genuinely distinct* concepts that merely correlate in this intake (`communal_social_orientation` ↔ `cooperative_values_alignment`, r=0.845; the `breadth_of_contribution_roles` meta-axis, ~0.79 with several axes). Merge correctly refuses these (different concepts), and it should — a private-but-values-aligned applicant proves they're distinct axes.

Auto-deactivating on correlation would attack the second with a tool that has no semantic check, and it fails three ways: (1) **correlation here is a 26-applicant sample artifact, not a property of the concepts** — it can vanish next intake, so deactivating a real axis on it discards a distinction the committee may need; (2) it **removes human agency silently** — the architecture keeps model output *inert until a human acts* (weight is 0 until a dimension is tiered), and auto-deactivation inverts that by making a weighting decision for the committee on grounds they can't see; (3) **double-counting can only occur when the committee actively tiers *both* correlated axes** — so the human is standing exactly at the decision point when it matters. That's where to *inform*, not preempt. The right shape (unbuilt, noted for later): surface a **redundancy hint at tiering time** — "you've weighted X and Y, which score applicants near-identically in this pool; you may be counting one factor twice" — reusing the `overlap` signal's math, leaving the choice to the committee (they may *want* both). The distinction in one line: **merge answers "are these the same concept?" (identity, system-decidable, safe to auto-apply); double-counting is "am I weighting one factor twice?" (a weighting judgement the committee owns).**


**The problem.** Re-ranking today is destructive to the committee's deliberation. `POST /rank/run` always calls `create_run()`, which inserts a *brand-new* `ScreeningRun` whose `criteria` has no `tiers` key — so `tiers()` falls back to the default layout (everything back in A-Tier, or under the new design, everything back in Ignore). Tiers are keyed to a run and `get_current_run()` returns the newest run, so the prior run's tier placements — which can represent real committee debate — are silently abandoned. And because the no-op gate only *allows* a re-rank once the pool has changed, the trigger is often tiny (one new applicant), yet the cost is total (re-discover every dimension, re-score the whole pool, reset every tier). This is the sharp edge we are removing.

**Why we cannot just match keys.** Dimension `key`s are LLM-generated free text, only *instructed* to be "stable snake_case." Discovery is a fresh synthesis call with no memory of the prior run, so a key recurring verbatim is luck, not a contract. Four cases arise across two discovery runs: (1) same concept, same key — lucky, transfers; (2) **same concept, drifted key** (`participation_commitment` → `stated_participation`) — a pure key-match silently *misses* it and drops a deliberate placement; (3) genuinely new dimension; (4) dropped dimension. Case 2 is the trap: it looks like the feature works (some keys carried) while quietly losing others. Naive string/normalized matching risks the worse failure — a *false* match that lands the committee's "S-Tier" intent onto the wrong concept. So we do not match on keys.

**The design — two LLM passes, then auto-applied carry-forward.** Re-rank stays a single button that always re-discovers, but the chain gains a second, narrow pass:

1. **Pass 1 — blind discovery (existing `discover_patterns`, unchanged).** It must *not* see the prior run's dimensions: showing them would anchor the model, make it reproduce old wording, and artificially inflate matches — defeating genuine re-discovery and causing us to over-trust the carry-forward. Blind discovery keeps both the discovery and the subsequent match honest.
2. **Pass 2 — identity match (new pass).** Given the old dimensions and the new dimensions (each as `name` + `definition` only — not keys, since keys carry no meaning), the model returns a mapping `{new_key → old_key | null}`, strictly one-to-one, **only at a very high confidence bar**. It does **not** discover or weigh anything — it answers a pure *identity* question ("is this new dimension the same concept as that old one?"). This is why it does not violate "AI discovers what varies, the human decides what matters": the importance decision (the tier placement) was already made by a human; the match pass only *recognizes identity* so that human decision can ride forward onto the semantically-equal new dimension. The mapping is stored as an audit artifact like every other AI output.

**High bar + auto-apply, no preview.** Because the match bar is set high — *exact intent match; when in doubt, return null* — we auto-apply the results with no confirmation screen, keeping the flow tight. The deliberate asymmetry: erring toward *under*-matching means a true survivor occasionally drops to Ignore and the committee re-promotes it (mild annoyance); erring toward *over*-matching would silently scramble intent onto the wrong concept (a trust failure). We lean to under-matching every time.

**Applying the mapping — three cases; "new" means *unmatched*, not *unplaced*.** After the match pass, each new dimension is one of:
- **Matched a prior dimension that was in a working tier** → carried into that tier. (Auto-applied.)
- **Matched a prior dimension that was in Ignore** → left unplaced (the committee's "ignore this" decision carries forward), but **not** flagged new — they already weighed in on it.
- **No match to any prior dimension** → left unplaced **and** added to `new_dimension_keys` (badged "New"), because it is a dimension the committee has never seen.

The critical distinction is **matched vs. not-matched**, *not* which tier the match landed in. An earlier bug branched on "did the placement land in a working tier?" — which conflated a genuinely-new dimension with one that matched a prior **Ignored** dimension (the latter's match-key isn't in the working-tier map, since Ignore isn't stored), so prior-Ignored survivors were wrongly badged "New" (e.g. `long_term_residency_intent`→`long_term_stability_intent`, and even byte-identical `financial_admin_skills`). Fixed by branching on `old_key is None` (no match) rather than on whether the carried placement found a working tier. There is still deliberately **no separate "New" bucket** (unmatched-new dimensions live in Ignore, weight 0, so they can't influence the ranking until triaged).

Why no separate "New" bucket at all: a second bucket has undefined semantics the moment the committee leaves something in it ("what do we do with un-triaged 'new' items?"). Putting unmatched dimensions in **Ignore** makes the answer well-defined and *correct by construction* — weight 0 means an un-triaged dimension cannot silently influence the ranking before anyone decided it matters, the same "human decides what matters" principle we protect everywhere. "Leave it alone and we ignore it" is then literally true, not a special rule.

**Ignore is the absence of a placement, not a stored tier (refactored after Phase 2).** The run stores only the *working* tiers; a dimension in no working tier has weight 0 — "ignored" by absence. The Ignore zone the tier-list UI shows is **synthesized for display** (`display_tiers`) from whatever is unplaced, and stripped again on save (`set_tiers` drops the incoming Ignore tier). This was chosen over modeling Ignore as a real tier (an `ignore: True` flag + a must-exist invariant): the explicit-tier version forced a "layout must contain an Ignore tier" rule, `non_ignore_count` filtering, and an Ignore-cloning dance in carry-forward — all to maintain a tier whose only meaning is "weight 0," which absence already expresses. The derived model deletes that machinery (no invariant: you can't omit a tier that isn't stored; unplaced→0 is the natural data model; carry-forward simply doesn't place unmatched dimensions). The frontend wire format is unchanged (it still receives and sends an Ignore zone). No backward-compatibility shim for the old explicit-Ignore stored shape: the local DB was reset instead (per the engineering rules — resetting dev data beats a tolerance read).

**"New" is a transient annotation, not a tier.** An unmatched dimension sitting in Ignore is **badged "new"** (amber chip + "New" pill) so it is silent to the *math* (weight 0) but loud to the *human* (clearly flagged as needing a decision). The badge shows iff its key ∈ `new_dimension_keys` **and** it is still in Ignore.

`new_dimension_keys` means "**unacknowledged** new dimensions," and a member acknowledges (clears the badge) **two ways**, both folded into the tiers `PUT` and recomputed in `set_tiers`:
- **Place it in a working tier** (drag to Critical/Important/Minor) — it's no longer unplaced, so it drops out; this is the "I've decided where it goes" path.
- **Acknowledge in place** while keeping it in Ignore — a per-badge **✕** ("mark reviewed, keep in Ignore") or a bulk **"N new · Mark all reviewed"** control on the Ignore row. This sends `acknowledged_keys` in the PUT; the keys drop out of `new_dimension_keys` but stay unplaced. This is the "I've reviewed this and it genuinely doesn't matter" path — a real decision the committee couldn't previously express (before, a deliberately-ignored dimension nagged forever).

So acknowledging = removing the key from the set, by either path; only **re-discovery** re-flags a dimension. This also fixes a subtle wrinkle of the older "badge = key ∈ set && in Ignore" rule: dragging a new chip to a working tier and back to Ignore used to re-show the badge (the set still held it); now placement removes the key, so it stays acknowledged. The tiers `PUT` response returns the recomputed `newDimensionKeys` so badges refresh in the same round-trip. One stored set per run, no per-chip "seen" flag.

**First run is not flagged.** The "new" badge means "appeared since you last tiered — you haven't weighed in yet." On a first run that is true of everything, so the signal would carry zero information. It falls out for free from the definition: `new_dimension_keys` is populated *only* by the match pass, and the match pass only runs when there is a prior run to carry forward from. First run → no match pass → nothing flagged → everything in Ignore, unbadged. The first-run empty state (below) already communicates "everything needs triage," so per-chip badges would be redundant noise.

**Default layout changes to "everything starts in Ignore" (both fresh runs and re-ranks).** Previously a fresh run started every dimension in A-Tier (= equal-weight baseline). Under carry-forward the consistent, honest default is **all dimensions in Ignore**, Critical/Important/Minor empty, for *both* a first run and the unmatched remainder of a re-rank. This inverts the failure mode from "we guessed your placements and might be silently wrong" to "nothing is placed; you decide" — appropriate because a re-rank is precisely when dimensions may have changed. It also means we never need to compare keys across runs to decide a *default*: matched dims are placed by the match pass; everything else is Ignore.

**Required math fix (landmine).** "Everything in Ignore" breaks `weights_from_tiers` as written: its uniform-fallback guard only fires when there are **zero non-ignore tiers**, but Critical/Important/Minor still *exist* (just empty), so the guard does not trigger — every dimension gets weight 0, every candidate's fit is 0, and the ranking collapses to an arbitrary `application_id` sort. The fix: change the fallback condition to **"no dimension received a positive weight → uniform weights"** (consistent with the existing "they cannot have meant rank-on-nothing" intent). Then an all-Ignore board ranks on the equal-weight baseline — identical to today's all-in-A behavior — until the committee tiers something.

**Required UI: baseline/empty-state label.** Because an all-Ignore board now *counts* (via the uniform fallback), "Ignore" momentarily does not mean "ignored," which would be a silent lie. The ranked view needs an explicit empty/baseline state when nothing is weighted yet — e.g. *"No criteria weighted yet — showing an equal-weight baseline. Drag criteria into tiers to weight them."* The "new" badges reinforce that there is triage to do.

**Per-dimension scoring — only send the model what actually needs scoring (cost — must land with this).** The core idea: **do not feed the model every dimension; feed it only the dimensions that are new or changed for that candidate, and reuse cached scores for the rest.** Today scoring is one call per candidate that hands the model the *entire* dimension set (`_dimensions_block` lists all of them), and the result is cached under `kind = dimension_scoring:<dims_hash>` where `dims_hash` hashes the whole sorted key set. Two consequences, both wasteful: (a) changing *any* dimension changes the whole hash, so **every candidate cache-misses and the full pool re-scores on every re-discovery**, even for unchanged axes; and (b) even within one call, the model generates a fresh score + rationale + evidence for *every* dimension, so output tokens (the dominant cost) scale with the full set every time. The fix has two halves that go together:
- **Cache scores per (candidate, dimension)**, not per (candidate, whole-set), so an unchanged dimension's score survives a re-discovery.
- **Narrow the prompt to only the to-score dimensions** for each candidate — the model is asked to score just the new/changed axes, not re-emit the ones we already have. Fewer dimensions in the prompt → fewer output tokens → real money saved. The cached scores are then merged with the freshly-scored ones to assemble the candidate's full `DimensionScoringReport`.

Which dimensions a candidate needs scored is known precisely: the **unchanged (matched) dimensions from Pass 2 are reused; only the unmatched/new ones are sent to the model.** So the match mapping does double duty — it carries tiers forward *and* tells the scorer exactly what to put in each candidate's prompt. (For the add-a-dimension fast-follow, the same machinery means a newly added axis sends *one* dimension to the model per candidate, not the whole set re-scored.) This is a real change to the scoring cache granularity (cache-key scheme), the per-candidate prompt construction (score a subset, not all), and the assembly step (merge cached + fresh), and needs its own design care when we build it; flagged here so it is not forgotten. It directly powers fast-follow #2.

**Resolved approach (decided at build time): per-dimension cache rows + one batched call per candidate.** Store scores at **per-(candidate, dimension)** granularity (so a matched dimension's score is reused precisely and survives across runs), but send the candidate's *uncached* dimensions to the model **together in a single call** — never one call per dimension, so the candidate's facts + essays are never repeated. This combines per-dimension reuse with one-call-per-candidate token efficiency. The cross-run cache identity of a dimension is supplied by the match mapping: a matched new dimension inherits its old counterpart's cache identity (→ hit, reused), an unmatched one gets a fresh identity (→ miss, scored). This is why score reuse (build Phase 4) depends on the match pass (build Phase 2).

**Estimator and confirmation screen must be updated.** The Rank cost estimate and confirmation card currently assume "discover + score the whole pool against a fresh dimension set." Carry-forward changes the real cost: discovery still runs, the new match pass adds a (small) cost, and scoring should only price the *uncached* (candidate, dimension) pairs once the per-dimension cache lands. The estimate, the cap check, and the confirmation breakdown all need to reflect this so the projected number stays honest (the project has already been bitten by an estimate that disagreed with the toast). Treat the estimator/confirmation update as in-scope for this fast-follow, not an afterthought.

**Open seams to settle at build time** (mechanical, noted so they do not surprise us): which model runs the match pass (likely the cheap first-pass/Haiku model — it is a short structured identity comparison, not synthesis); whether the match-pass cost folds into the existing Rank estimate and the single cap check (expected yes); the exact "new" badge visual treatment; and the precise shape of the per-dimension scoring cache key and prompt. **We expect to discover more as we get into it** — this is a genuinely cross-cutting change (run model, a new AI pass + schema, the weight-fallback math, scoring cache granularity, estimator, and several UI affordances), so the build should start by re-reading this section and re-confirming the seams before writing code.

#### Committee-Proposed Criteria (propose)

A fast-follow on carry-forward that lets the committee *steer what discovery looks for* without violating "AI discovers what varies; the human decides what matters" — proposing an axis is asking the model to *check whether the pool varies on it too*, still discovery, not a weighting.

- **Proposed** — free text a member writes ("school-age kids who'd use the playground"). Persists on `criteria.proposed_dimensions` (survives reload), fed to the next Rank's discovery prompt as **"strongly consider, but you decide"**, **consumed** by the next run (it becomes a real dimension), then cleared.

The model may refine wording, split one request into several, merge overlapping ones, or **skip** a requested axis if the pool genuinely doesn't vary on it. It flags every dimension it creates from a request with `PoolDimension.from_committee_request` (retained as provenance — it feeds `reconcile_audit` and lets the UI show an axis originated from a member ask; the flag survives the model renaming/splitting it, which a text-match never could). Discovery is therefore **blind except for the committee's proposals**: with no proposals it is fully blind (the original first-run behaviour); proposals steer only the axes the committee named. Persistence is `PUT /screening/seeds`; the Rank confirmation card notes how many proposals it will offer.

**Propose is complementary to automatic reconcile (below), not redundant with it.** Propose = a member names a *brand-new* axis for discovery to consider; reconcile = the system silently re-checks *historical* axes. Different inputs, different initiators, no overlap. A member never needs to re-propose an axis the committee saw before — reconcile re-checks all history automatically.

**Favourite is KEPT — removal REVERSED 2026-07-10.** An earlier decision (2026-07-09) slated the **favourite** seed (★ on each criterion, "keep this axis across re-runs") for deletion, on the reasoning that *automatic reconcile subsumed it*. (This removes the "favourite removal" item from 4c's cleanup sweep.)

**RESOLVED + BUILT 2026-07-11 — favourites and proposals are routed DIFFERENTLY, by what each needs.** The pre-planned option (c) — "inject *both* favourites and proposals at decomposition, discovery fully blind" — was refined during implementation once we separated the two asks by their needs. They are opposite cases:
- **A favourite is a prior dimension.** It already has a pool-grounded definition and cached scores and was already shown to differentiate. It needs nothing from discovery except a **guarantee it stays on the table**. → **Injected at decomposition** (all K discoverers stay blind), flagged `from_committee_request`, where `enforce_committee_requests` (the D9 guard, extended to take `favourites`) guarantees survival deterministically. The settling call, seeing every carving at once, folds any re-discovered twin INTO the favourite (reusing its key → match adopts it → cached scores carry forward).
- **A proposal is an untested free-text hypothesis** that has never touched the pool. This is exactly where discovery's real job — grounding vague text in applicants' own words, sharpening it into a measurable axis, and gating on pool variance — matters most. → **Seeded into ONE discoverer (worker 0)**, not all K. This preserves K−1 independent blind samples (diversity/coverage integrity intact) while grounding the proposal and keeping a soft pool-variance gate (worker 0 can decline a flat proposal). It then flows through decomposition/match/D9 like any discovered axis.

**Why the split beats uniform (c):** (c) rejected "seed one discoverer" (option b) because a single-report axis might not survive decomposition. That risk is real for a *favourite* (whose survival must be certain) but acceptable for a *proposal* (an untested hypothesis that *should* be gated out if the pool is flat — declining it is correct, not a failure). So favourites get the certain path (decomposition + D9 backstop) and proposals get the grounding path (worker 0). Only the favourite gives up the pool-variance gate, which is exactly right: **Jeff accepted this (2026-07-11)** — for a favourite the whole point is "keep it even if discovery would drop it," so gating defeats the purpose; a flat axis is ~0 impact anyway. A proposal keeps a soft gate, because grounding-then-gating is the whole reason a raw hypothesis wants discovery.

(2) `discover_patterns_fanout` renders the pool digest once, then composes a *seeded* prompt (worker 0, carries proposals) and a *blind* prompt (workers 1..K−1) — identical when there are no proposals, so the un-seeded path is unchanged. (3) `dimension_decompose.build_prompt`/`decompose_dimensions` take `favourites: list[PoolDimension]`, rendered in a `<favourite_axes>` block that instructs "each MUST survive, reuse its key, fold twins in." (4) `enforce_committee_requests` takes `favourites` and folds them into its never-vanish ask set by key (flag-loss repair + silent-drop re-add + folded-list), so a favourite is guaranteed even if the model omits it. (5) `do_criteria` resolves favourites to their prior `PoolDimension`s and passes them to both `decompose_dimensions` and the guard; proposals go to the fan-out as before. Auto-favouriting still fires via the `from_committee_request` flag riding through. Editing either prompt (discovery/decompose) shifts its PROMPT_VERSION → the rank-inputs fingerprint restales, so this change busts the cache and the next Rank re-runs fresh (expected). Tested: a favourite reaches the decompose prompt (not discovery); an unsurfaced favourite is re-added by the guard; a proposal reaches exactly one of K discoverers.

#### Automatic Reconcile Of Dropped Dimensions (resolved 2026-07-09)

**Resolves the parked open question "surface all previously discovered dimensions?"** The underlying committee need is narrow: *"a factor we care about didn't show up in this run's chips, but it may still matter for this pool — don't silently lose it."* The chosen mechanism is **discover-blind-then-reconcile**: keep blind discovery pure, then automatically re-check dropped priors against the live pool, with the **model** (not the member) deciding whether each still applies. This beats both parked options — a browsable "menu of all prior dimensions" (lets a member re-assert a flat axis by hand, violating "AI decides what varies") and "just lean on propose" (makes the committee remember and re-type history the system already has).

**Vocabulary — two distinct concepts, two words (don't conflate).** They overlap but neither contains the other, so one word can't carry both:
- **Revived** (member-facing) — a dimension that was **absent from the prior run** but is **present now**: it disappeared from the committee's view and came back. This is a **presence gap**, and it is **mechanism-agnostic** — it does not matter *how* the dimension came back (blind discovery re-finding it, or the reconcile pass recovering it); the member only experiences that it left and returned. Drives the **badge**.
- **Recovered** (observability) — the **reconcile pass** specifically pulled an axis back that *this run's blind discovery had missed*. This is a **mechanism** event, invisible to the member, and is what the audit watches to catch over-recovery (RQ2's rationalization risk). Drives the **audit**.
- Why they must differ: blind discovery re-finding a gapped axis is **revived but not recovered** (no reconcile involved); the reconcile pass salvaging an axis that was in the *last* run but discovery dropped *this* run is **recovered but not revived** (no presence gap — the member never lost it). See RQ5/RQ8.

Requirements (locked):

- **RQ1 — Automatic, all-history, invisible.** Every re-rank, *after* blind discovery + the match pass, the system reconciles all-history dropped priors (`all_known_dimensions − matched-this-run`) against the live pool. No member action; no browsable history menu. Members only ever see axes the model re-confirmed.
- **RQ2 — The model decides, with stronger-than-discovery discipline.** For each dropped prior, the model answers a pure "does the live pool *vary* on this?" question; a flat axis is a **no**, and **"no" is the expected answer for the majority.** This is the rationalization guard: "what about these?" biases a model toward yes (the same overconfidence the match pass showed on drifted concepts), so the reconcile prompt must carry a *harder* bar than discovery, not a softer one.
- **RQ3 — Blind discovery stays uncontaminated.** Reconcile is strictly a downstream second look — in *shape* **the match pass in reverse** (match: "is this new dim the same as a prior one?"; reconcile: "here are priors that didn't reappear — do any hold here anyway?"), running once per re-rank on the same cheap model tier. **But not the same economics** (correcting an earlier claim in this section): the match pass is a pure text-vs-text *identity* compare that never sees the pool (~2k tokens total), whereas reconcile must answer "does the **pool vary** on this?", which is unanswerable from dimension text alone. So reconcile takes a **pool-sized input** — see the design below for exactly what it reads.
- **RQ4 — A revived axis restores its prior placement; the badge is the safeguard.** Any axis absent from the prior run but present now (a *revived* axis, per the vocabulary above) restores its **last committee tier placement** (all-history, via `tier_history`) and takes that tier's weight immediately. It is **badged "Revived"** so the committee sees it re-entered weighting after a gap and can veto/re-tier. (Chosen over hold-at-weight-0-until-acknowledged and revive-only-to-Ignore: consistent with carry-forward's durable-intent principle; the brief window of influence-before-notice is accepted, mitigated by the badge.)
- **RQ5 — Badge taxonomy is presence-driven, not mechanism-driven.** The badge tracks *the member's experience of a dimension's presence across runs*, not which internal pass surfaced it:
  - **No badge** — present in the **prior run** (continuous in the committee's view). How it surfaced *this* run is internal plumbing: whether blind discovery re-found it or the match pass carried it, it never left, so nothing to flag.
  - **"Revived"** — **absent from the prior run** but present in **some earlier run**: a presence gap, now closed. **Mechanism-agnostic** — a gapped axis re-found by *blind discovery alone* (no reconcile) still badges "Revived," because the member experiences the same "it came back." (This also retroactively flags a case that used to resurface *silently*: all-history match — SPEC.md carry-forward note — could already resurrect a gapped dimension with no badge; under this rule that correctly reads "Revived.")
  - **"New"** — never appeared in **any** run: parked in Ignore at weight 0, needs full triage.
  The internal mechanism (discovery-match vs. reconcile-recover) does not vanish — it lives in the audit (RQ8), it just stops driving the *badge*.
- **RQ6 — "Revived" clears like "New."** Acknowledgeable by placement, per-badge ✕, or bulk-clear; only re-reconciliation re-flags.
- **RQ7 — Revival is partially cached, not fully fresh.** A revived axis **adopts its historical key**, so unchanged candidates previously scored on that key are cache **hits**; only new/edited candidates score fresh. (This corrects the earlier parked pessimism that revival "scores fresh (cache miss)" — that was true only under whole-set caching, which Phase 4's per-(candidate,dimension) cache replaced.) Whatever *does* miss must appear in the Rank estimate/cap — no phantom "free" revival.
- **RQ8 — A `reconcile_audit` view is required from day one.** Sibling to `match_audit`: per run, how many dropped priors the reconcile pass was *offered* vs. how many it *recovered* (the mechanism concept — see Vocabulary, distinct from the member-facing "revived" badge). Makes over-recovery visible and testable — the requirement-level version of the skeptic check, operationalizing "no is the expected answer for most" rather than leaving it to prompt wording alone. Two logging requirements make the audit actually usable rather than aspirational (both cheap now, painful to reconstruct later — and consistent with "raw AI analysis/traces accessible," SPEC.md):
  - **RQ8a — Log the reasoning, not just the count.** Store the model's short justification for *each* reconcile decision, so a spot-check can see *why* an axis was recovered without re-running the pass.
  - **RQ8b — Log rejections, not just recoveries.** Over-recovery shows up in what was recovered; *under*-recovery (the mild, accepted failure mode from RQ4's asymmetry) shows up only in what was *declined*. Logging only the yeses is blind to half the behavior.

**Architecture — single deterministic call first; escalate to multi-agent only if observability shows over-recovery (decided 2026-07-09).** Build the simplest thing: **one deterministic reconcile call**, slotting cleanly into the existing rank-chain seam (its input is pool-sized, not match-sized — see RQ3 and the design below). **Model — synthesis tier (Sonnet), not the cheap model (decided at build, 2026-07-09).** An earlier note here guessed the cheap first-pass model; the build corrected it against the match pass's own history: the match pass was *tried* on Haiku and **over-matched** on exactly this kind of "is this the same/does this apply" judgment, so it was moved to Sonnet. Reconcile carries the *identical* rationalization risk (RQ2), and over-recovery is the specific failure the audit exists to catch — so it starts on the trusted model rather than knowingly repeating the match pass's mistake and betting the audit catches it. `reconcile_model` is Admin-configurable; move it cheaper if the audit shows Sonnet is overkill for this narrower question. (This is not at odds with measure-first: the ladder is about *judgment machinery* — single-call vs. multi-agent — not about starting on a model a sibling pass already proved too weak.) *Then measure, before spending on orchestration* — the same measure-first stance M6/M7 take on model choice. The escalation ladder, cheapest signal first:
- **Observability (free, always on — RQ8).** `reconcile_audit` logs offered-vs-recovered every real run. If the pass recovers the *majority* of dropped priors, RQ2 already tells us that's a bug signal — the gross over-recovery failure is caught by the rate alone, no labeled data needed.
- **Eval (built only if observability warrants).** The rate can look sane (e.g. 3 of 20) while the model recovers the *wrong* 3 — reviving flat axes, declining ones that genuinely vary. Catching that needs a small **hand-labeled eval set** of `(pool, dropped_prior) → should-recover?` pairs scored offline for precision/recall. It costs real labeling work, so it is deferred until a spot-check (enabled by RQ8a/RQ8b) suggests the *judgment*, not just the rate, is off.
- **Multi-agent (the escalation, not the default).** Only if eval shows the single call genuinely over-recovers under RQ2's rationalization pressure do we reach for the multi-agent shape — a blind discovery agent, N independent reconcile judges, and an adversarial skeptic that defaults to "no" and must be argued out of it (which makes RQ2's discipline *structural* rather than prompt-dependent). It buys independence at more tokens and orchestration cost, and would be the project's first real multi-agent loop — a legitimate learning investment, but one we let the data justify rather than assume. (Terminology note: "eval" = offline scoring against a labeled ground-truth set — *is the judgment correct?*; "observability" = always-on instrumentation of real runs — *is it behaving weirdly?*. They answer different questions; the ladder uses the cheap one first.)

**The build design (single-call v1, mapped onto the existing rank chain).** The criteria phase already runs on a worker thread as **discovery → match → `adopt_matched_keys` → `carry_forward_layout` → `create_run`** (`app/api/ranking.py`). Reconcile is a **new step inserted between `adopt_matched_keys` and `carry_forward_layout`**, so a revived dimension re-enters `report.dimensions` under its historical key *before* carry-forward runs — which means tier restoration (RQ4) and per-dimension score reuse (RQ7) both happen through the **existing** machinery, by plain key equality, with no change to `carry_forward_layout` or the scoring cache. New module `app/ai/dimension_reconcile.py`, mirroring `dimension_matching.py`'s layout (module-level `SYSTEM_PROMPT` + `_INSTRUCTIONS`, `derive_prompt_version`, a `build_prompt`, an `estimate_*`, and one entry function). Concretely:

- **What it's fed (settles the "same economics" correction).** Two inputs in one call: (1) the **dropped priors** — `all_known_dimensions(db) − new_to_old.values()`, i.e. every historical dimension the match pass did *not* map a fresh dimension onto (a set difference over data already computed in the setup block at `ranking.py:357-360`; no new query); and (2) a **compressed pool view** — the *same per-candidate digest discovery already builds*. Discovery's `_candidate_digest` (`pattern_discovery.py`) emits `{applicant_id, facts, essay_analysis (minus evidence)}` per candidate and already proves the whole pool fits one call; reconcile reads the identical view. **Settled: extract `_candidate_digest` into a shared helper both passes call** (the same anti-drift discipline `applicant_facts.py` enforces for discovery/scoring — one serializer, provably identical pool view), and reuse discovery's proven **~600 input-tokens/candidate** budget rather than guessing. Compressed, not raw essays: a genuine fresh look at the pool (so it can catch variation discovery missed — RQ1) at O(pool)-once input, ~$0.10–0.20 per re-rank, without re-paying discovery's full raw-essay read. It does **not** see this run's freshly-discovered dimensions (that would just re-assert discovery's own conclusion); it sees the *pool* and the *dropped priors*, and judges independently.
- **Schema — a verdict per offered prior** (`ReconcileReport` / `ReconcileVerdict` in `app/ai/schemas.py`). Unlike `DimensionMatchReport` (matches only, omissions implicit), reconcile returns **one entry for every dropped prior**: `{old_key, revive: bool, reasoning: str}`. This directly satisfies RQ8a (reasoning on every decision, both ways) and RQ8b (rejections are explicit, not inferred) — the audit sees the full ballot. Bounded cost: the pass runs once per re-rank, and a reason string per "no" is affordable at that frequency. Result is **sanitized** like the match pass: drop unknown/duplicate `old_key`s; a `revive: true` on a key not in the dropped set is ignored.
  - **Start rich, downgrade once trusted (reversible knob, not a redesign — decided 2026-07-09).** Ship the **full ballot** (reasoning on every verdict, both ways) from day one: at the start we most need to *see* what the pass does, and the extra output tokens buy exactly the spot-check visibility RQ8a/RQ8b exist for. Once `reconcile_audit` shows the pass is "doing pretty good" over real runs, the reasoning-on-declines can be dropped to trim output cost — degrading to the **revivals-only + derived-rejections** shape (rejections = offered − recovered by set-diff; reasoning only on `revive:true`). This is the **mirror of the escalation ladder**: that ladder spends *up* on judgment machinery when trust is low; this spends *down* on audit richness when trust is earned — same observability-vs-cost tension, opposite direction. Keep it a schema/prompt knob (drop the required-reasoning-on-decline instruction; the set-diff for rejections already exists in the view), so it's a cheap dial to turn either way if a later prompt change reopens doubt — never a rebuild.
- **How a revival takes effect.** Each `revive: true` verdict pulls that prior dimension (key + latest text, from `all_known_dimensions`) **back into `report.dimensions`** before carry-forward. From there it is indistinguishable from any other carried key: `carry_forward_layout` restores its most-recent tier placement (RQ4), per-(candidate,dimension) scoring reuses cached scores under the key and only scores the coverage holes (RQ7), and `create_run` persists it. No revival-specific path through carry-forward or scoring — reconcile's *only* job is to decide which dropped keys re-enter the report.
- **The RQ2 prompt bar.** The `SYSTEM_PROMPT`/`_INSTRUCTIONS` carry a *harder-than-discovery* discipline (not the match pass's identity bar): the question is strictly "does the **live pool vary** on this axis?" — a flat axis is a **no**, an axis that doesn't apply to this cohort is a **no**, and **"no" is the expected answer for the majority** (these are precisely the axes fresh discovery already declined to name). Frame it as skeptical review, not "what about these?", to fight the rationalization pull. Carries the `INJECTION_GUARD_NOTE` like every pass that reads laundered human text.
- **The badge is derived from presence, and reuses the ONE existing flags set — no parallel structure (settled, per engineering rule "before adding a parallel data structure, ask whether existing data can carry the need").** Reconcile emits `recovered` (mechanism); the member-facing `revived` **badge** is a presence-gap fact (RQ5): a key **absent from the prior run but present now** (whether blind discovery *or* reconcile brought it back). Two temptations to resist, both parallel structures:
  - *Don't* store a separate `revived_dimension_keys`. "New" and "Revived" are the **same kind of thing** — an *unacknowledged flag* on a chip that needs triage (RQ6: "Revived clears like New") — differing only in **label + color**, and that difference is **derivable from run history**, not a fact to persist. So keep the single existing flags set (today's `new_dimension_keys`, which already carries persistence + the acknowledge machinery + the `TierList → TierRow → DimensionChip → ChipBody` threading) and **derive the label per chip at read time**: `tier_history` already walks all runs newest-first, so it cheaply also returns the **immediately-prior run's key set**; then for a flagged key, `revived` = `key ∈ known_keys` (seen before) AND `key ∉ immediately_prior_keys` (gone last run); `new` = `key ∉ known_keys`. One stored set, one threading path; amber-vs-blue is a history lookup.
  - The only genuinely-new *behavior*: a re-surfaced key must now **enter** that flags set (today `carry_forward_layout` adds only never-seen keys). That is a one-line broadening of the membership condition — `key ∉ immediately_prior_keys` instead of `key ∉ known_keys` — not a new structure.
- **One uniform acknowledge rule — no per-badge branch (settled 2026-07-09).** Both badges are the *same* mechanism (an unacknowledged flag); the acknowledge rule is identical for both, and the New-vs-Revived difference is just *where each kind of chip starts*, not a second code path. The rule: **a flag clears on a member action — the explicit ✕ / "mark reviewed", or a member dragging the chip into a tier — but carry-forward's *auto*-placement is not a member action, so it does not clear.** That single rule produces both behaviors for free:
  - *New* keys are never auto-placed (carry-forward has no prior placement for a never-seen key → they start in Ignore), so the only way a New chip becomes placed is a member dragging it — which clears it. "Placement clears New" is thus not a special case, just the rule applied to a chip that only ever sees member placement. (Existing behavior, unchanged.)
  - *Revived* keys **are** auto-placed (RQ4 restores the prior tier), and since auto-placement isn't a member action, the flag persists — so a revived-into-Critical dim keeps its blue badge until the committee explicitly reviews it. That is the RQ4 safeguard: a revived dim silently sitting at max weight is the *most* important one to surface. **All revived dims are flagged regardless of where they land.**
  So there is no `if new/revived` branch in `set_tiers`' recompute — one flags set, one clear-on-member-action rule; the label is derived for display only. The invariant the implementation must preserve: **auto-placement never clears a flag; a member action (drag or explicit acknowledge) always does.**
- **Badge visual (settled).** Sibling to the existing "New" chip: `TierList.tsx` already threads `isNew?: boolean` → a `tier-chip-new-badge` span; add `isRevived?` → a `tier-chip-revived-badge` span the same way (the boolean is the derived label, computed where `newKeys` already is). Color: **"New" stays the strongest alarm (amber — never triaged, always in Ignore); "Revived" is a softer heads-up (blue/info — seen before, just came back, may already be placed)**, reusing the `impact` score-band palette's existing meanings (strong→green, moderate→**blue**, weak→amber, SPEC.md) rather than inventing a color.
- **Audit — `reconcile_audit`, sibling to `match_audit_view`.** Persist the full ballot on the run's `criteria` (offered priors, each verdict + reasoning, the compressed-pool candidate count), and expose a `reconcile_audit_view` in `ranking_run.py` mirroring `match_audit_view`: offered count, recovered count, **recovery rate** (recovered / offered — the over-recovery smell, RQ8), and the per-prior reasoning for spot-checks. Surfaces in the same admin trace view as the match audit.
  - **Zero-recovery still writes a row (settled).** Follow `match_audit_view`'s precedent exactly: it already distinguishes a *first run* (rate `None` — N/A, nothing to match against) from a *genuine zero-match re-run* (rate `0.0`). Same here — the audit row is written **whenever the pass runs**; `None`/absent means the pass was *skipped* (first run / no dropped priors). A zero-recovery run is the *healthiest* RQ8 signal ("no is expected for the majority," often for *all*), and RQ8b's "log rejections" is precisely a full ballot of reasoned no's — exactly the record worth spot-checking, never suppressed.
- **Estimator + cap.** Fold reconcile's projected cost into the combined Rank estimate and the single cap check, **only when a prior run exists** (first run has no dropped priors — pass skipped, like the match pass). Because its input is pool-sized (not match-sized), estimate it as **`~600 tokens/candidate × pool size`** (reuse discovery's proven `_DISCOVERY_INPUT_TOKENS_PER_CANDIDATE`, since the pool view is now literally the same shared digest — seam 1) **+ a per-prior output allowance × dropped-prior count** (the full ballot's reasoning-per-verdict, so output scales with the offered set, not the recovered set). *Not* the match pass's flat 2k/600 constants. (Same "measure the real cost shape of the call" lesson as the scoring-estimate fix below.)
- **Prompt version → fingerprint.** Like the match pass, reconcile is uncached but its `derive_prompt_version` output must fold into `rank_inputs_fingerprint` (`ranking_run.py`), so editing the reconcile prompt correctly marks Rank "out of date".

**Seams settled 2026-07-09** (were flagged open; now resolved, recorded above): (1) **compressed-pool serialization** → extract and share discovery's `_candidate_digest`, reuse its ~600 tokens/candidate budget (no separate serializer, no drift); (2) **zero-recovery audit row** → always written when the pass runs, `None` only when skipped, following `match_audit_view`'s first-run-vs-zero precedent; (3) **"revived" badge** → sibling `isRevived?` chip in blue/info (softer than "New" amber), plus the `carry_forward_layout`/`tier_history` change to compute the presence gap against the immediately-prior run's key set. **OPEN — locked-pool accumulation: the experiment, then a possible efficiency redesign (parked 2026-07-09, MUST RUN before building anything here).** Reframing from a build-time realization: the SPEC treats a re-rank as "the pool changed, reconcile the survivors," but the *dominant real-world case is the opposite* — **the applicant pool is locked** (real screening happens after submissions close; only preliminary ranking runs on a moving pool). In that dominant case, re-running Rank is not change-handling — it is a deliberate strategy to **harvest discovery's nondeterminism into the "fullest set" of dimensions** for the committee to weigh. This is philosophically clean (accumulating axes over the *same* locked data characterizes that one pool more completely; it is NOT the "generic ideal member" drift, which only applies *across different pools*). But it raises three unresolved questions, and **we must run the experiment before designing for it** (the discipline: data first, per the real-data rule):

- **The experiment (do this first, on real Bedrock): does the set converge?** Lock a pool, run the chain ~5× unchanged, and measure: (a) does the distinct dimension-key count **converge or creep** (the 67-keys-for-~22-concepts failure was *supposedly* fixed by all-history matching but **never validated to actually stabilize**); (b) does discovery's **split/merge/reword churn** settle, or oscillate in granularity (the match pass is strictly 1:1 — it structurally cannot represent "one concept became two," so a `financial_stability` ⇄ `income_level`+`income_resilience` wobble may never converge); (c) does reconcile's skepticism **help (filters spurious splits/noise) or hurt (thrashes, declining legitimate axes)**.
- **Reconcile's skeptic bar is calibrated backwards for this case.** RQ2's "*no* is expected for most" is right when the pool *changed* (a dropped axis may reflect a real change). But on an **unchanged** pool, a dimension discovered in run 1 provably still varies (identical data) — a run-2 non-appearance is *pure nondeterminism*, so skepticism will wrongly decline legitimate axes and defeat accumulation. The signal to tell the cases apart is **already in hand — the pool fingerprint**; a pool-aware reconcile (skeptical when the pool changed, **generous when it did not**) is one prompt branch on a boolean we already compute. **Jeff's lean: generous accumulation when the data isn't readily available to decide otherwise** — but confirm against the experiment before hard-coding it.
- **Efficiency redesign — decouple accumulation from scoring (LEADING candidate, but GATED on the experiment).** Running the *whole chain* K times to sample discovery is wasteful: scoring is the expensive phase (O(candidates × dimensions), whole pool) and discovery is one cheap call (~$0.10), yet the chain welds them, so 5 re-runs pay for **5 full pool-scorings just to make discovery talk 5 times**. The efficient shape for a locked pool: **fan discovery out K times (parallel, cheap, no scoring) → union/dedupe into the stable set (the identity/match problem, run over K samples instead of across runs) → score ONCE against the settled set** — roughly a ~5× cost cut on the dominant path, since scoring dwarfs discovery. This is *exactly* the "first multi-agent workflow" already parked (K discovery agents + a union/reconcile step + an adversarial skeptic to reject noise); the efficiency instinct and that parked note are the same design arriving from two directions, and fan-out *also* narrows reconcile's job from "argue a subtle axis back under rationalization pressure" to the cleaner "dedupe K samples and reject noise." **Gate: build this only if the experiment shows accumulation genuinely stabilizes and is worth optimizing.** It does *not* help the pool-*changed* case (scoring must re-run there — but that case is rare), and it does not by itself answer convergence (cheaper sampling makes the experiment cheaper to run, not already-answered).
- **Two watch-items regardless of path:** (a) **split-concept double-weighting** — if the fullest set lands both `income_level` and `income_resilience` in a tier, that concept gets ~2× the weight of one that stayed atomic, quietly distorting the deterministic ranking math; (b) **member-added dimensions are the stable anchor, not a confounder** — they carry a durable key by construction so they don't churn, but a member axis near-duplicating a later discovery-found one hits the same double-weight risk.

**Scoring-estimate fix (carry-forward skew).** Per-dimension scoring rows store a batched call's tokens **split evenly across the dimensions that call scored** (`_split_usage`) — correct for cost accounting, but it poisons the cost *estimate*. A scoring call's **input** is ~95% shared overhead (the candidate's facts + essays, sent once per call regardless of dimension count), so a carry-forward re-run that scores a single uncached dimension attributes the whole ~2.9k-token prompt to one row — dragging the learned per-row input average up ~10× and ballooning the ceiling estimate (observed: a phantom $1.35 that tripped the cap). Fixed by estimating the **real cost shape of a call**: per-CANDIDATE input measured from a real built prompt (~chars/4, immune to the split) + per-DIMENSION output learned from the rows (output *is* genuinely per-dimension) × dimension count, priced one call per candidate. Output stays self-tuning; input no longer does (it can't, given the split). This is a concrete instance of why M13 (observability/evals) matters — the skew was invisible until a real run surfaced it.

**The decision and its evidence.** The locked-pool convergence experiment (n=10; see the reconcile "Findings" block + close-out ledger, and `docs/case-studies/dimension-convergence.md`) proved that re-running the Rank chain to accumulate the "fullest set" of dimensions **does not usefully converge** — discovery re-carves the same concepts at different granularities each pass (participation sliced 8-9 ways), and the sequential machinery (match 1:1 + reconcile) hoards every carving. The root cause: reconcile's per-axis **variance test is near-unfalsifiable** (every axis discovery coins *does* vary), so coverage/redundancy — not variance — is the only discriminating question, and that question can only be answered with **all carvings visible at once**. Hence fan-out.

**The shape.** **K parallel discovery calls (no scoring) → one decomposition step that sees all K reports at once and settles the finest non-overlapping set → score ONCE against the settled set.** Cheaper than K full re-runs on the dominant locked-pool path — but see the cost note below: the win is real, though NOT for the reason first assumed. The decomposition step is the crux — see the "reframe" below for what it must actually produce (NOT "one grain per concept," the tempting-but-wrong first framing that the session-2 reread corrected).

**Cost model — corrected from the real ledger (2026-07-09 s2; earlier "scoring dwarfs discovery" claim was WRONG).** Queried `run_cost_ledger` for the 10 runs. Per-unit: one (dim × candidate) score = **$0.00087**, one discovery call ≈ **$0.17**, one full settled-set scoring pass (≈22 dims × 27 cand) ≈ **$0.52**. So at K=5 the fan-out splits **K discovery ≈ $0.85 vs. scoring ≈ $0.52 — discovery is the BIGGER half**, because discovery is uncached every run while essays + most scores are cache hits. Consequences: (1) my "scoring dwarfs discovery, so K is ~free" intuition is falsified — **K carries a real linear cost** (K=5 nearly doubles discovery spend vs. today's single call), which *strengthens* D6's fixed-small-K decision and suggests **K=4** as the default over 5. (2) The scoring term is the only one with a `dims × candidates` multiplier, so it's the one that grows with pool size — at 300 applicants the settled set costs ≈$5.75 vs. ≈$9.15 for the 35-dim bloat, a $3.40 gap that is pure dimension-count waste. So the redesign attacks the right cost (fewer dims → less scoring) — the payoff is modest at n=27 (total ≈$1.37/run) but scales with the pool. (3) Jeff's "score-everyone-is-prohibitive" worry: not true at n=27, directionally right at scale.

**The reframe that reset the design (2026-07-09, session 2 — re-read all 10 runs in the DB, not just the case-study summary).** The union step is NOT "pick one grain per concept" — that framing would flatten genuinely-distinct axes (a nurse's `health_safety_skills` and a treasurer's `professional_financial_management` are different people at the top; collapsing them robs the committee of separate levers). Reading every run's carvings showed the territory holds **two different things at once**: real re-carvings of ONE axis (`participation_commitment_depth` / `governance_participation_commitment` / `maintenance_participation_commitment` — same "will you show up," sliced by domain → collapse) *and* legitimately distinct axes that must stay apart. So the target is **the smallest set of axes that are each genuinely differentiating AND mutually non-overlapping** — a *decomposition* problem (factor the territory finely without overlap), not a *selection* problem (choose one carving). This is the load-bearing correction; every decision below follows from it.

**The empirical keystone — run 10 proves the question becomes falsifiable when all carvings are visible.** The case study framed the failure as one-directional (reconcile revives everything because "does it vary?" is unfalsifiable — discovery only coins an axis *because* it saw variance). But run 10's recovery rate finally fell to **56%**, and its declines switched from variance to **redundancy** reasoning on their own: `community_program_enrichment` *"overlaps substantially with community_program_skills and participation_commitment_depth — redundant"*; `community_organizing_experience` *"largely already captured by governance_body_experience"*. Once the pool was saturated enough that overlaps sat in one view, the model spontaneously became **able to say no on redundancy grounds**. That is exactly what fan-out formalizes: showing all K carvings at once turns "is this redundant with what we already have?" into a decidable question — the case study's core lesson ("the architecture determines whether it's asked something answerable") made concrete. It also confirms the tension is **two-sided**, killing the earlier "over-merge is the only risk" framing: **under-merge** = today's creep (weight participation 9×); **over-merge** = collapse the nurse into "contribution" (lose a real lever). Both are live; the design must guard both.

- **O2 — the pass/fail metric, made operational.** "Finest stable non-overlapping set" decomposes into three *measurable* properties, all computable by `analyze_convergence` from data we already persist:
  - **Non-overlapping — measured from cached scores, NOT by asking the model.** The real test ("would these two definitions score the same applicant the same way?") needs the model, but the metric must not consult the same model it's judging (it would grade its own homework). The instrument-grade proxy is already on disk: the per-(candidate, dimension) 0..1 scores (`ApplicationAIResult.output["score"]`, read by the exact join `candidate_scores` uses). Two dimensions that overlap show highly **correlated score vectors across the pool** → compute pairwise Pearson correlation over candidate score vectors; flag any surviving pair above a threshold (start **r > 0.9**, tune on real data) as a suspected un-merged overlap. Deterministic, reuses existing data, independent of the judge. (Caveat to check when building: a pair can correlate for a real reason — e.g. two genuinely-distinct skills both track "high-agency applicant" — so a high-r pair is a *flag to inspect*, not an automatic fail; the score-alike *definition* is the ground truth, correlation is the cheap detector.) **Timing gotcha (do NOT wire correlation into the decomposition loop):** score vectors exist only *after* scoring, but the fan-out scores once against the *already-settled* set — so a freshly-minted axis has no scores at the moment the Merger is deciding. Correlation is therefore an **offline judge metric** (decompose → score → `analyze_convergence` computes overlap), not an inline signal the Merger can consult. The Merger must reason from definitions + evidence in the moment; correlation grades the result afterward. (For the bake-off on the locked 10-run pool this is a non-issue — most axes already carry cached score vectors from prior runs via matched keys.)
  - **Finest — most axes under the overlap ceiling.** Once overlap is detectable, "finest" is the largest set that stays clean: score `≈ distinct_count − penalty × overlapping_pairs`. 25 axes all r<0.9 beats 22; 25 with three r>0.95 pairs is worse than 22 clean. This is what makes the metric *directional*, not just a pass/fail — the creep (35 dims, many overlapping) scores badly on both terms at once.
  - **Stable — same K inputs settle to the same set (this IS O3's measurement).** Run the decomposition M times on the **same** K discovery reports; measure set agreement across runs. Compare by **concept clustering, not raw keys** (keys are minted per run, so raw-key Jaccard would understate agreement — map each output axis to its nearest prior-run axis by the same score-vector correlation, then Jaccard over the clusters). High agreement = stable; the bake-off winner must be both finest AND stable.
- **O3 — the decomposition step's own nondeterminism (folded into O2's stability term).** The step is itself a model call (or a loop), so running it twice on the same K reports may settle differently — O2's stability metric measures exactly this. If measured variance is high, that is a **finding**, and it feeds back into D7's bake-off: a deterministic-clustering pre-pass (cluster similar axes by score-vector correlation *before* the model decides grain) re-enters as a way to stabilize — the option D7 currently sets aside. So O3 isn't a separate task; it's the reason the stability term exists, and the escape hatch if the pure-LLM decomposition proves too jittery to trust.

### API Redesign (Milestone 11)

**Goal: redesign the HTTP surface to best practices — the "if we could go back and design it from scratch, what would we change?" pass.** Taken now on purpose: the feature set is essentially complete (M1–10 done, M9 fast-follows done), and crucially **there is no public API contract to break** — the only client is our own first-party frontend, which we change in the same commits. That makes this the rare safe window to fix the surface without a deprecation dance. Once a second consumer or a public contract exists, this becomes expensive-to-impossible; doing it while the cost is near-zero is the whole point.

**Scope: the API layer only.** This is a *contract* refactor — routes, verbs, status codes, payload shapes, error format, naming — not a rewrite of the screening logic underneath. The services (`screening_run`, the AI passes, the import pipeline) and the data model are out of scope except where the API forced an awkward shape onto them. The frontend (`api.ts` and its callers) moves in lockstep, in the same commits, since it is the sole client.

**Step one is an audit, not edits.** The current surface grew organically across ten feature milestones (~21 endpoints across `applications`, `auth`, `dashboard`, `quality_flags`, `screening`, `settings`, `sync`, `health`), so before changing anything the milestone catalogs every endpoint and names where it diverges from REST/HTTP convention. Known smells to evaluate (not a predetermined fix list — the audit confirms or rejects each):
- **Resource modeling & naming** — are routes resource-oriented and consistently named? (e.g. action-shaped paths like `/rank/run`, `/seeds` vs. resource nouns; `screening` as a catch-all router holding tiers, seeds, ranking, estimate, run.)
- **Verb & status-code correctness** — are GET/POST/PUT/PATCH/DELETE used per their semantics; are 200/201/202/204/4xx/409/422 returned meaningfully? (e.g. the run endpoints that stream, the estimate endpoints, the `409`-on-no-eligible pattern.)
- **Payload consistency** — request/response shapes, camelCase-vs-snake_case discipline across the boundary (the codebase mixes both today), envelope vs. bare-array responses, pagination shape.
- **Error format** — is there one consistent machine-readable error contract, or ad-hoc `detail` strings per route?
- **Long-running / streaming operations** — the NDJSON Rank stream and the cost-estimate-then-confirm pattern: is there a clean, consistent convention for "estimate → confirm → run → stream progress"?
- **OpenAPI quality** — does FastAPI generate a clean, well-typed schema (tags, response models, examples), and should response models be tightened?

**Process, given it's a major change.** Audit → propose the target surface (a written before/after map of every endpoint) → get sign-off on the shape before editing → migrate route-by-route with the frontend moving in the same commit → keep the test suite green throughout (the API tests are the safety net that lets us refactor confidently). Reset the local DB rather than write compatibility shims if a payload change touches stored shapes (per the engineering rules). Because there's no contract to preserve, prefer clean breaks over backward-compatible layers.

**Decisions locked (2026-06-26):**
- **camelCase everywhere on the wire** — request bodies, response bodies, and query params. Enforced by direction-split Pydantic alias bases in `app/schemas/base.py`: `ResponseModel` (serializes to camelCase), `RequestModel` (accepts ONLY camelCase — snake_case input is rejected, so the contract is enforced not merely tolerated), and `BridgeModel` (accepts both + emits camelCase, the deliberate exception for `AppSettings`, which is also reconstructed from the snake_case JSON stored in `admin_settings.value`). One generator, no per-field aliases, no recursive response camelizer (which couldn't tell a field *name* to rename from a snake_case *value* like a dimension key to leave alone). Domain dataclasses (`app/domain/ranking.py`) and storage schemas (`app/ai/schemas.py`) stay pure — casing lives only in boundary `*Out` response models.
- **RFC 9457 problem+json for every error** — one machine-readable shape (`{type, title, status, code, detail, instance, ...extensions}`, media type `application/problem+json`), replacing ad-hoc `{detail: "string"}`. A single `Problem` exception + a code→(status, title) registry in `app/api/problems.py`, rendered by two handlers registered in `main.py` (`Problem` and FastAPI's `RequestValidationError`, so framework 422s share the contract). The frontend branches on `code`.
- **Rename `/screening` → `/ranking`** — the router *is* the ranking subsystem (rank chain, ranked list, tiers, seeds, run criteria) and the UI calls it "Rank"/"View ranking"; "screening" is the app's umbrella term, so it was actively misleading. The redundant `rank/` segment collapses (`/screening/rank/run` → `/ranking/run`). `/quality-flags` is NOT renamed to `/screening` in M11 — it would collide with the `ScreeningRun` model (which means *ranking*). That two-way rename (`/quality-flags` → `/screening`, `ScreeningRun` → `RankingRun`) rides with **M12**'s DB refactor, when renaming the model frees the word "screening" with no interim collision. (`/sync` is deliberately *kept*, not renamed to `/import`: the operation is an idempotent upsert-by-email reconcile, which "sync" describes more precisely than "import", and the codebase already speaks sync throughout — `SyncRun`, `source_sync_run_id`, `isSyncing`. The UI's step was relabeled "Sync" to follow the system, rather than renaming the endpoint to follow a button.)

**Why before observability (M13).** Observability instruments the API (per-endpoint metrics, audit surfacing) and a second screening actor (M14) is a new consumer of it — both are cheaper to build on a clean, consistent surface than to retrofit, then re-instrument, after a later redesign. Sequencing the redesign first avoids doing the wiring twice. (The DB schema refactor lands as M12 in between — also cheaper to do once the API contract above it is settled.)

### Observability And Evals (Milestone 13)

The pipeline now makes real, non-deterministic model judgments (discovery, scoring, the carry-forward match pass), and we have already hit the wall their opacity creates: when a re-rank carried 18/18 organic dimensions forward by identical key, we could not tell *genuine re-discovery* from *match-pass over-matching* — because the stored run kept only the rewritten result, not what discovery actually emitted or how the match pass reasoned. That gap is the motivation: **M13 makes the AI legible — what it cost, what it did, and whether it is any good.** The first brick already exists (`criteria.match_audit`: raw pre-adopt discovery dims + the new→old match map + the match narrative); M13 generalizes that instinct across all four pillars.

Behavior-neutral: no DB failure record, no logging *destination* config, no `ItemErrorEvent` wire change (all still deferred — see Pillar 3). This was the true prerequisite for Pillar 3's failure rates.
1. **Per-pass AI trace viewer (Pillar 2, reframed — see below).** Make every pass's raw output legible, match-audit included as one panel — not the whole stage.
2. **Cost surfacing (Pillar 1) — DONE (2026-07-08; unified 2026-07-12).** A "Cost" Insights subtab (`CostPanel` / `GET /ranking/insights/cost` + `/last-runs` / `cost_report.py`) showing both **cumulative** AI spend across all runs and the **last Screen/Rank**, broken down by pass with a per-pass token (in→out) + model breakdown. **Unified cost accounting (2026-07-12).** Every pass — pool-level (discovery, decompose, match, consolidate) and per-application (screening, scoring) — now returns one shared `PassCost` value object (`app/ai/pricing.py`: calls, input/output tokens, cost, cache side, model), built once from a call's `Usage` so tokens are never discarded. `record_run_cost` writes them to a single **`RunPassCost`** table (one row per pass, under a thin `RunCostLedger` header per run) that **both** Screen and Rank write to and both cost surfaces read. (b) *Every pass attributed separately* — discovery/decompose/match/consolidate/scoring each record their own row (a pass that made no call this run records a zero row, so the pass set always covers `RANK_PASS_LABELS` by construction — no drift). NOT surfaced: estimate-vs-actual reconciliation (deferred — now unblocked, since the exact per-run figure to compare against exists).
3. **Operational metrics (Pillar 3) — BUILT (2026-07-12).** An Insights "Trends" subtab (`MetricsPanel` / `GET /ranking/insights/metrics` / `services/metrics.py`) charting per-run and per-pass trends across all completed runs: cost, tokens, wall-clock latency, cache-hit rate (over cacheable units only, so a never-cached pass doesn't drag it to 0), failure count, and — for Rank — live dimension count over time. Latency correlates ledger→`RankingRun` by creation order (1:1, same request; no FK). Also the surface a later LLM-judge score would accrue on (see the judge docket item's "watched as a trend" dependency).
4. **Property-based evals (Pillar 4) — BUILT (2026-07-12; deterministic layer).** `app/evals/` scores a committed fixture (one blessed real Rank, recorded via `python -m app.evals.fixture`) against checks split into two kinds by whether they can *honestly* pass/fail deterministically:
   - **INVARIANTS** (hard-fail pytest, the CI gate — `tests/test_evals.py`): things that are ALWAYS a bug — `poles_present` (every dimension has distinct high/low ends; the mechanical form of the household-size direction punt) and `no_protected_attributes` (no criterion keys on a protected class). These are **discovered-green**: they pass on the fixture because the output is genuinely good, never because a check was tuned to the data.
   - **SIGNALS** (report-only via `python -m app.evals.run`, never block): judgement calls a human must make — `overlap` (dimension-pair score correlation ≥ threshold: escaped duplicate *or* legitimate confound?) and `match_rate` (carry-forward %, flag only 0/100% degenerate ends). Forcing these green would just teach us to weaken the check.
   - **Key discipline (this is the transferable lesson):** the split is by *determinism, not by "is it an eval"* — the industry anti-pattern is putting fuzzy/nondeterministic checks in the commit gate. A check you'd have to soften to keep green is a signal, not an invariant (`overlap` started mis-filed as an invariant with an "examined-not-exists" tautology, and moved). **Prompt versioning is derived, not hand-bumped (foundation for evals).** Each cached pass's `prompt_version` is a hash of its *static* prompt text — system prompt + instruction template + any shared fragment it embeds (`app/ai/prompt_fragments.py` guardrails; `FILTERED_FACTS_NOTE`) — computed once at import via `derive_prompt_version(...)` and threaded into `cache_key`. Editing a prompt therefore changes only that pass's version and re-runs only its cache, automatically; there is no global version constant and nothing to remember to bump. (This replaced an earlier global `PROMPT_VERSION = "N"` that bumped *all* cached passes for any one prompt's edit and relied on a human remembering to bump it correctly.) The static/dynamic split is deliberate: applicant content is already covered by `raw_row_hash`, so only the *template* is hashed. Per-application values are interpolated at call time, outside the hash. **Per-settings values that change the model's answer are the exception — they belong in the version.** The pet-policy threshold is such a value: screening's version is `screening_prompt_version(settings)`, which folds in the *filled* pet-policy line, so changing max cats/dogs misses the cache and shows Screen "out of date". (An earlier design kept it out — "just a threshold it cites" — which was a bug: a policy change reused stale results and reported "up to date" while over-limit applicants stayed wrongly flagged. Fixed 2026-07-08; guard: `test_screening_version_changes_with_pet_policy`.) Pattern discovery and dimension matching are uncached and have no version. This matters for M13 because honest evals require attributing every stored result to the exact prompt that produced it: with derived versions, an eval set is automatically partitioned by prompt revision, and a prompt change can never silently reuse results judged under the old wording.

**Pillar 2 — Per-pass AI trace viewer (match-audit included).** *Reframed 2026-07-07:* the goal is legibility across **all five passes** — what each one actually output — not just match. Match-audit is one panel inside this, not the whole stage. Two honest axes, because the passes store their output differently:

- **Per-application × pass** (the three cached passes — screening, essay analysis, dimension scoring — each writes `ApplicationAIResult` rows with `output`/`narrative`/tokens/cost/`model_id`/`prompt_version`): this legibility belongs **on the candidate detail page**, co-located with the candidate (per the "per-item legibility lives on the item" decision above). Two of these are **partly done already** — the detail page renders the screening narrative + flags and the essay-analysis structured output as *product* surfacing (findings, cleaned up), reading the very same rows a trace would. So the work here is **extend in place, do not build a parallel viewer**: (a) add trace metadata (tokens/cost/`model_id`/`prompt_version`) to the two existing sections; (b) add the one missing pass — **dimension scoring** has no raw surfacing today (per-dimension rationale/evidence). Building a separate per-application viewer that re-renders the narrative/essay output would duplicate the display and drift — the detail page is the single source of truth for per-application AI legibility.
- **Per-run** (the two uncached passes — discovery and matching — live in `run.criteria`, no per-application row): this has **no home today** and is the genuinely new surface. Belongs on the ranked view: discovery's raw report, plus `match_audit` (raw-discovery vs. survived, the map + narrative, and the **carry-forward rate** = matched / total). **[UPDATED 2026-07-11 — the "high rate = smell" framing is reversed, see below.]** Originally the near-100% rate was read as an over-matching smell; under the fan-out redesign a high carry-forward rate is *expected and good* (the settled dimension set has stabilised run-to-run). The aggregate rate can't distinguish "correctly stable" from "wrongly matched", so it is no longer framed or coloured as an alarm; over-matching is watched via a WRONG match in the ballot/narrative, not the rate.

> **SURFACED (Stage 1 run-level axis — done 2026-07-07).** `match_audit` is no longer a SQLite spelunk. `GET /ranking/current/match-audit` returns the raw-discovery dimensions, the new→old map (each matched new-key resolved to the prior dimension's **key + user-facing name**), the match narrative, and the derived counts + **carry-forward rate** (`match_audit_view` in `ranking_run.py`); the `MatchAuditPanel` renders it, colouring the rate amber (≥60%) / red (≥90%) toward the over-matching band, and showing both the friendly name and the system key on each side of a match. Two small captures were added at write time (where `prior_report` is in hand): `prior_dimension_count` (so the rate is null on a genuine first run rather than guessed from an empty map) and `prior_dimension_names` (so the "carried forward from" column shows the prior title, not just its key). Runs written before a given field render its fallback until re-ranked (no shim, per the disposable-DB rule): "first run" for the missing count, "(prior dimension)" for a missing name. Verified live against real Bedrock: two consecutive re-runs sat at ~71% and 73% carry-forward. **REFRAMED 2026-07-11:** those numbers were originally read as "healthy churn, not over-matching" — but the over-matching-smell framing was a sequential-era artifact. Under fan-out + decomposition the match pass runs on the *settled* set, and a HIGH carry-forward rate is the goal (dimension space has stabilised, re-ranks reuse tiers + scores). So the panel no longer colours the rate as an alarm (amber/red bands removed for match; the shared CSS stays for the reconcile over-recovery rate, where high IS still the smell). A genuinely bad match is a *wrong* mapping — caught by eyeballing the ballot, which the aggregate rate can't reveal. (The residual real risk from SPEC line ~525 — a wrong match now reuses a cached *score*, not just a tier chip — is exactly why the per-match ballot, not the rate, is the watch surface.)

The split, per Jeff: **applicant-specific** audits go on the candidate detail page (co-located with the candidate they explain); **general/run-level** audits go on the Insights tab. Insights is gated on a run existing (like Ranking) and is the home where Stage 2 (cost) and Stage 3 (metrics) will also land — not a one-off for the match audit.

Exposed via a `discovery_narrative` field on `GET /ranking/current` (no new endpoint). **Principle (Jeff, 2026-07-07): observability must not duplicate product content — it's only for what has no home in the product.** So the discovered *dimensions* aren't repeated here (each dimension's **why-it-differentiates** was added under its description on the Ranking tab, tap a criterion), and neither is the *pool summary* (already the Ranking tab's "Show more" blurb). The reasoning narrative is the only genuinely-observability-only artifact, so it's all that remains. The per-application axis of Stage 1 is therefore intentionally empty — the passes' *findings* are already on the candidate page (flags, essay summary, fit dimensions), and their *cost trace* belongs in aggregate, not per-candidate. (Note: this removal makes the earlier Stage-1-COMPLETE claim moot; Stage 1's live surfaces are the run-level Insights subtabs.)

**Pillar 3 — Operational metrics.** Pipeline health, not output quality: run counts, per-pass latency, cache hit rates (we already isolate cached vs. fresh in `RunTally`), dimension counts over time, and failure/retry rates per pass.

> **Prerequisite: AI-call failures are currently invisible — fix capture before measuring rates.** The "failure/retry rates per pass" above assume failures leave a trace; today they don't. When a model call fails, `run_in_pool` isolates it to `(item, None, exc)`, `screen_applications` flattens it with `error=str(error)` (losing the exception *type*), and it streams once as an `ItemErrorEvent`, then is discarded. There is **no logging configured anywhere in the backend** and **no failure record in the DB** (`ApplicationAIResult` stores successes only — a failed call writes no row). So we cannot answer "how often, and which failure mode?" — we have exactly one anecdotal observation (a ~2-min stall during scoring that self-healed, consistent with the 120s Bedrock `read_timeout` working as designed; boto3 already retries throttling/5xx/timeouts with `max_attempts=5, adaptive`). Note the inconsistency to resolve: the ranking criteria path already preserves the type (`type(exc).__name__: {exc}` at the `StreamErrorEvent`), while the screening path does not.
>
> **Scoped first step (behavior-neutral observability) — DONE 2026-07-07:** (1) `error_type` added to `PassResult`, sourced from a shared `exception_type_name` helper (fully-qualified class); (2) emitted durably via stdlib `logging` on the error path in the screening (`analysis.py`), dimension-scoring (`dimension_scoring.py`), and ranking-criteria (`ranking.py`) paths. No migration, no behavior change; logging *destination* left unconfigured (deferred, below). After a few real runs the failure-mode distribution is known, and *then* the retry decision is evidence-based (the leading candidate being a narrow retry of Pydantic schema-validation failures — nondeterministic and often parse on a second attempt — explicitly NOT throttling, which boto3 already owns).
>
> **Decisions deferred (pick when this pillar is built):** logging vs. a durable DB failure record (note: a DB record is more invasive than "add a column" — failed calls write no row today, so it changes the storage/cache model); if logging, destination (stdout vs. a file under `data/`) and matching the run/launch convention; and whether to carry `error_type` onto the streamed `ItemErrorEvent` for live visibility.

> **Open question to answer here: why do some streaming calls emit zero deltas?** During the streaming work we observed (and a fan-out spike reproduced: 42 of 50 short Haiku calls) that some calls return the correct structured output + usage but stream **no** reasoning deltas at all — while others on the same path stream hundreds. The leading hypothesis is that short/fast calls under concurrent load come back as a single chunk rather than an incremental stream, but we haven't confirmed it. It's **not a correctness issue** (output and token accounting are always right), purely a UX/observability curiosity — the criteria phase's live "thinking" panel is occasionally silent. Worth instrumenting (delta count per call, by model/length/concurrency) when this pillar is built, to confirm the cause and decide whether it's worth smoothing.

**Pillar 4 — Quality evals (property/consistency first).** BUILT 2026-07-12 — see the numbered Pillar 4 entry above for the shipped design (invariant/signal split, PII-safe fixture, discovered-green discipline). The original plan called for one-concept heuristics and score-stability too; the build revised that from evidence: **one-concept was cut** (an "&" in a name is a conventional single concept as often as a bundle — a string check can't tell "Health & Safety" from a real bundle, so it's a semantic judgement deferred to LLM-judge), and score-stability deferred (wants repeated real runs). What shipped as hard invariants: schema/poles conformance and no-protected-attributes. Overlap and match-rate ship as report-only signals. Golden-set and LLM-as-judge remain the explicit next layers.

**Surface (revised 2026-07-07 — see "Locked plan" above).** The original "dev harness first, then admin UI" ordering is kept only for **Stage 4 evals**, where a replayable CLI/pytest harness genuinely unblocks prompt tuning and CI. Stages 1–3 go **in-app first** and are **ungated** (no admin view — that gate waits for M14). Per-application trace legibility (Stage 1) extends the candidate detail page in place; per-run audit, cost, and operational metrics are aggregate surfaces (ranked view / a metrics view). Recorded-output fixtures matter: evals should be replayable against captured real responses so the suite runs in CI without burning Bedrock spend on every run — the structured `output` (and `narrative`) is preserved per call for exactly this. **Correction (2026-07-07): the input *prompt* is NOT stored** — `ApplicationAIResult` keeps `input_tokens` but not the prompt text (earlier "persists input/output… the eval substrate" phrasing overstated this). For a trace viewer that's fine (prompts rebuild deterministically from application + `prompt_version`); for eval *replay* it means Stage 4 must either rebuild the input from the application + version or add input capture. Decide at Stage 4 which; don't assume the input is already on hand.

**Sequencing note.** This is a milestone-sized, cross-cutting build (new persistence/aggregation, an eval harness, an admin UI surface). It should be planned as its own slice with its own checkpoints, not bolted onto a feature in flight. The `match_audit` capture landed early only because a live diagnosis needed it; treat that as the seed, not the start of the full build.

### Agent Workflow

MVP implementation should bias toward simplicity, readability, and understandability over maximum agent sophistication.

The application is *designed as* a multi-agent system, but the agents are a conceptual model, not a mandate to build orchestrated LLM loops everywhere. The roles below are real and useful as a decomposition; how each is *implemented* depends on whether it benefits from coordination.

**Realized architecture (through milestone 7): a pipeline of single-purpose passes + human gating.** Each "agent" so far is a named, user-visible pass — deterministic code (hard filters) or one structured-output call (screening flags, essay analysis, pattern discovery, dimension scoring). State lives in the database between passes; the "orchestration" is the human clicking gated workflow steps in order plus deterministic control flow (`screen_applications` fanning out a batch, the dimensions-hash threading discovery into scoring). No LLM decides what runs next, and no agent calls another agent. This is deliberate, not a shortcut: the product's hard requirements — pre-run cost estimates and a spending cap, per-(candidate, kind, prompt-version) caching, auditability of every prompt/output/rationale, eval-replayable units, and reproducible structured output — all depend on the call graph being known in advance. A swarm whose calls are decided at runtime would fight all five. **Screening is a known, ordered task, so a pipeline is the right spine — not a fallback from a swarm.**

**Where genuine multi-agent coordination earns its place.** A `Coordination Agent` that plans work, routes between agents, and decides when an output needs revision before the user sees it *is* wanted — but only where the task has a feedback/revision loop or a runtime-variable call graph that a fixed pipeline cannot express. Those spots are specific, and coordination should be added there surgically (and stay **bounded** — generate→critique→retry-N, not open-ended), never as a swarm replacing the pipeline:

- `Evidence Auditor` (M8+): the first real loop. After ranking/recommendation, an auditor checks each recommendation is grounded in the application data and sends weakly-supported ones back for revision — a bounded generate→critique→maybe-regenerate cycle. This is the cleanest first place a Coordination Agent supervises a revision loop.
- **M9 re-weighting loop**: drag a criterion into a tier → re-sort → adjust again. By design the re-sort is deterministic math over cached scores (see M7), so the "loop" is human-in-the-loop + arithmetic, not an LLM orchestrator spending tokens per turn — there is no model call at all in M9 as shipped. (The future Criteria Coach fast-follow adds a *bounded* model touch: it reflects on the weighting the committee built and challenges it, but still does not re-rank.)
- `Screener-Evaluator` (M7+): evaluates the *system across runs*, not one candidate, and proposes human-approved, versioned improvements (a new named schema field + prompt-version bump, a prompt edit, a model swap). Loop-shaped but deliberately human-gated; schema/prompts are **never self-modified at runtime** — autonomous mutation would break comparability, caching, eval consistency, and auditability.
- A `Coordination Agent` becomes worthwhile once two or more of these loops run in one screening session and their order/retries need supervising (e.g. score → audit → re-score the flagged few → re-rank). Until then a coordinator over single-shot passes would be ceremony. Introduce it when the loops exist to coordinate, and have it orchestrate *bounded* sub-agent calls with explicit retry limits and a recorded decision trail — preserving the cost/audit/eval properties the pipeline gives us for free.

The full role set (each a pass today, some gaining loops later): `Ingestion Agent` (read rows, map columns, detect drift), `Hard Filter Agent` (deterministic eligibility, auditable reasons), `Essay Analyst` (M6, neutral extraction — does not judge fit), `Pattern Finder` (M7, discovers differentiating dimensions), `Dimension Scorer` (M7, scores candidates on those dimensions), `Criteria Coach` (M9 fast-follow, reflects on and challenges the committee's tier weighting — does not elicit or re-rank), `Ranking Agent` (M8–9, deterministic weighted-sum re-rank on the tier weighting), `Evidence Auditor`, `Report Agent` (M10, MOMI summaries/justifications/caveats), `Screener-Evaluator`, and the `Coordination Agent` above.

Every AI recommendation should be reviewable and overrideable. AI outputs should explain why a candidate advanced, not just provide a numeric score.

### Privacy, Auditability, And Evals

It is acceptable to send full application context, including names/contact context, to the AI model for this project. Redaction is not required for the initial design.

Applicant data should still be treated as sensitive personal information. The app should keep deterministic filtering separate from AI-assisted judgment, preserve auditability for prompts, model outputs, filter decisions, ranking rationales, and user overrides, and avoid writing back to source Google Sheets unless explicitly approved.

The app should include an eval-oriented design from early development:

- Maintain small fake fixture sets for deterministic-filter and AI-schema tests
- Test deterministic filters with unit tests
- Test AI output schemas for consistency
- Track whether agent recommendations are grounded in source fields
- Track hallucination, unsupported-claim, and evidence-quality failures
- Preserve enough trace data to debug regressions when models/prompts change

An explicit eval dashboard is not required for MVP, but the architecture should make it possible later.

Synthetic/sample applications are not required for MVP. Demos may use real co-op data only with people who are authorized to view it.

## Multi-Member MOMI Workflow

Each MOMI member should be able to run their own screening process separately. For each member, the app should preserve:

- The patterns surfaced by AI
- The questions asked by the app
- The member's answers
- The criteria or values inferred from those answers
- The resulting ranked shortlist
- Applicant-specific rationale and evidence

Each MOMI member should use their own login/name. Members may see each other's criteria and shortlists; anonymization is not required.

The app should then support combining member shortlists into a merged ranked list. The merged comparison should make criteria visible, so committee discussion can address not only which applicants were selected but why each member valued particular signals.

Merged shortlist behavior:

- Member rankings are weighted equally.
- Applicants appearing on multiple member shortlists should be prioritized.
- Applicants with strong disagreement, such as one member strongly rejecting a candidate while another ranks them highly, should be flagged for discussion.
- The merged output should include consensus recommendations, disagreement/discussion-needed candidates, and not-recommended candidates.
- A single-shortlist appearance may be shown if useful, but criteria comparison is more important than simply noting that an applicant appeared on one list.

Criteria comparison should identify:

- Criteria shared across committee members
- Criteria that differed between members
- Criteria unique to a member
- Plain-language summaries of each member's priorities, such as "prioritized practical co-op labour and clear participation commitment"

Members can write private, per-applicant notes separate from AI-generated rationale. Notes belong to their author, stay out of AI inputs and shared reports, and never appear to other members; they remain on the author's printed candidate detail.

For now, any user may finalize the merged recommendation list. In practice this is the MOMI chair's job, but a special chair account/role is not required for the initial design.

The committee-facing report should include both interview recommendations and a summary of how the pool was screened.

## Users, Roles, And Authentication

The MVP should support real Google login because multi-member screening is a major design requirement. For early MVP/testing, the app may run locally and Jeff may use multiple Google accounts to test authentication and role behavior.

Preferred authentication is Google login.

Access should be invitation/approval based when the app is live. Jeff is the initial admin and can invite MOMI members.

Roles:

- `Admin`: the initial account; can invite/manage users once invitations exist.
- `Member`: a MOMI committee screener — can run screening sessions, run shared cached AI screening runs, answer AI questions, rank candidates, add notes, and participate in merged comparison.

Every committee member is a trusted screener, so **the screening workflow has no admin-only surface.** Status overrides, the raw source row, and the raw AI narrative are all available to any logged-in member — these are just the source and reasoning behind data members already see, and the privacy boundary is screeners-vs-outsiders, with members inside it. The `Admin`/`Member` distinction exists in the data model (the first user created becomes admin) and is intended to gate user management when invitations are built, but it does not currently gate any route. Settings are login-only, not admin-only. The engineering default is `require_current_user`; add a role gate only for a genuinely admin-only capability, as a deliberate decision.

AI screening results are shared across users and cached per application content, model, and prompt version. Any logged-in member may run the checks; the cost-control concern is uncached work, not which member initiates a shared run.

A special MOMI chair/finalizer role is not required for the initial design.

Users may create multiple runs for the same application pool, such as "Jeff first pass" and "Jeff revised after thinking."

Screening runs should preserve enough source information to understand what application pool was used. The app does not need immutable snapshots; a sync/run metadata record is sufficient for MVP and likely sufficient long-term.

When criteria are revised after a completed run, the default behavior should be to update/overwrite the same run. The user should also be able to choose to create a separate new run instead.

## Data Storage

Recommendation:

- Use Google Sheets as the external source of truth for submitted applications.
- Import application rows into the app database for screening runs, AI outputs, user answers, notes, rankings, criteria summaries, and reports.
- Use SQLite for the local MVP because it is simple, inspectable, and reliable for one machine/testing.
- Design the data layer so it can move to a hosted database for multi-user use.

The expected live committee size is about 5 MOMI members. They may sometimes use the app concurrently, but heavy simultaneous usage is unlikely. A hosted database will eventually be needed for true multi-user use across computers.

The app should minimize repeated spreadsheet access by importing/syncing rows and then using the app database for screening state. Google Sheets should be accessed for refresh/sync, not for every AI or UI operation.

Core data model decisions:

- An `Application` represents one household/application.
- Application data includes applicant, co-applicant, children, essays, references, income, pets, declaration, source metadata, and screening metadata.
- Preserve the raw Google Sheets row exactly as imported alongside normalized fields.
- Store the raw Google Sheets row as JSON in SQLite.
- Use the primary applicant email address as the primary application identity, while still giving each stored application an internal database ID for relationships and history.
- Normalize email identity by trimming whitespace and lowercasing before duplicate detection or primary-key use.
- If Google Sheets contains duplicate emails, use only the last-added row.
- If a source row changes after import, update the existing application when the user explicitly syncs and then re-run screening logic when the user tells the app to do so.
- Duplicate detection for MVP is based on email address.
- The newest application wins by default for duplicate email addresses.
- Compute normalized fields during import/sync, including `adult_count`, `child_count`, `children_under_18_at_move_in`, `has_real_estate`, `household_income`, `pet_count`, and `pet_types`.
- Income parsing is straightforward because the form uses whole-number validation. The field always contains a clean integer for new submissions.
- Each sync should create a `SyncRun` record with timestamp, source sheet ID, row count, duplicate count, imported count, updated count, eligible count, filtered-out count, and needs-review count.

Admin settings such as Google Sheet link or ID, the hard-filter thresholds (income band, min/max children, max child age, pet limits), and the AI spending cap should live in the database rather than `.env`.

Local `.env` files should be supported for secrets and local credentials.

Use `.env.local` for local secrets and `.env.example` for safe placeholder values.

The following must not be committed to the repo:

- `.env` files
- OAuth credentials/secrets
- SQLite database files
- Applicant exports
- AI traces
- Raw prompts or model outputs containing applicant data

The repo should include a `.gitignore` before implementation.

## Reports

1. **Discovery doesn't split a *direction-contested* axis into two — it picks a direction and annotates its doubt instead (observed 2026-07-09, run 1).** The pool varies on pets (none → single cat → single dog → dog+cat, plus one "not sure yet"), and the split guidance intends **two** dimensions here (more-pets-is-better *and* fewer-pets-is-better, committee picks).
